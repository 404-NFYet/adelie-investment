### G-2. 호가창 시뮬레이션 (가짜 주문장)

**파일**: `fastapi/app/services/orderbook_simulator.py` (신규)

거래량/변동성 기반으로 매수-매도 스프레드와 잔량을 생성:

```python
class SimulatedOrderBook:
    """거래량/변동성 기반 호가창 시뮬레이션"""

    def generate(self, current_price: int, volume: int, volatility: float) -> dict:
        """
        Returns:
            {
                "asks": [{"price": 50100, "quantity": 342}, ...],  # 매도 호가 5단계
                "bids": [{"price": 49900, "quantity": 287}, ...],  # 매수 호가 5단계
                "spread_bps": 20,  # 스프레드 (bps)
            }
        """
        # 스프레드: 변동성 높으면 넓게, 거래량 많으면 좁게
        base_spread_bps = max(5, min(50, int(volatility * 100 - volume / 100000)))

        # 호가 단위: KRX 가격대별 호가 단위 적용
        tick = self._get_tick_size(current_price)

        # 잔량: 거래량의 0.5~3%를 5단계에 분배 (정규분포)
        total_depth = int(volume * random.uniform(0.005, 0.03))
        # ... 5단계 분배 로직
```

### G-3. 주문 타입별 체결 로직 개선

#### 시장가 주문 (기존 강화)

```python
# 호가창 시뮬레이션 → 주문이 잔량을 소비하며 체결
orderbook = SimulatedOrderBook().generate(current_price, volume, volatility)
if trade_type == 'buy':
    # 매도 호가(asks)를 낮은 가격부터 소비
    executed_price, filled_qty = _fill_against_book(orderbook['asks'], quantity)
else:
```

- 체결가 = 가중평균 (여러 호가 단계를 걸쳐 소비할 경우)

#### 지정가 주문 (신규 체결 추적)

현재 문제: pending 저장 후 방치. 해결 방안:

**즉시 판정 + 만료 방식** (실시간 트래킹 대신):

```python
async def execute_limit_order(db, trade, target_price, current_price, volume, volatility):
    orderbook = SimulatedOrderBook().generate(current_price, volume, volatility)

    # 1. 즉시 체결 가능 여부 (호가창 기준)
    if trade_type == 'buy' and target_price >= orderbook['asks'][0]['price']:
        return _execute_immediately(...)  # 지정가 이하로 체결
    if trade_type == 'sell' and target_price <= orderbook['bids'][0]['price']:
        return _execute_immediately(...)  # 지정가 이상으로 체결

    # 2. 즉시 체결 불가 → 확률적 체결 판정
    fill_probability = _estimate_fill_probability(
        target_price, current_price, volume, volatility
    )
    # fill_probability 계산:
    #   - 가격 차이가 작을수록 높음 (목표가 ≈ 현재가 → ~80%)
    #   - 변동성 높을수록 높음 (가격이 흔들리므로)
    #   - 거래량 많을수록 높음
    #   - 상한: 95%, 하한: 5%

    if random.random() < fill_probability:
        # 체결 성공 — target_price로 체결
        return _execute_at_target(trade, target_price, ...)
    else:
        # 미체결 — pending 상태 유지, 24시간 후 자동 만료
        trade.order_status = 'pending'
        trade.expires_at = datetime.now(KST) + timedelta(hours=24)
        return {"order_status": "pending", "message": "주문이 접수되었습니다. 24시간 내 체결되지 않으면 자동 취소됩니다."}
```

**만료 처리** — 기존 스케줄러에 추가:

```python
# scheduler.py 또는 별도 cron
async def expire_stale_limit_orders(db):
    """24시간 경과 미체결 주문 자동 취소"""
    expired = await db.execute(
        update(SimulationTrade)
        .where(SimulationTrade.order_status == 'pending')
        .where(SimulationTrade.created_at < datetime.now(KST) - timedelta(hours=24))
        .values(order_status='cancelled')
    )
```

## Part H: 챗봇 vs 에이전트 캔버스 UX 구분

### H-1. 현재 혼란 지점

| 구분 | 튜터 챗봇 (TutorModal) | 에이전트 캔버스 (AgentCanvasPage) |
|------|----------------------|-------------------------------|
| 진입 | ChatFAB / 용어 클릭 | AgentDock → router → `/agent` |
| UI | 오버레이 모달 (85vh) | 풀페이지 대체 |
| 응답 | 대화형, 간단 마크다운 | 구조화 마크다운 + 액션 버튼 그리드 |
| 컨텍스트 | 선택적 (일반 질문) | 필수 (페이지/종목 맥락) |
| API | `/tutor/chat` (plain) | `/tutor/chat` (canvas_markdown) |
| 세션 | 자동 저장 | 핀 → 리뷰 카드 |

**핵심 문제**: 같은 `/tutor/chat` API를 공유하면서 다른 UX를 제공 → 사용자가 두 경험의 차이를 이해하기 어려움.

### H-2. 제안: 역할 기반 시각 구분

#### 방법 1 — 캐릭터/아이콘 차별화

| 요소 | 튜터 챗봇 | 에이전트 캔버스 |
|------|----------|--------------|
| 아이콘 | 펭귄 (현재 그대로) | 펭귄 + 안경/돋보기 오버레이 |
| 이름 | "AI 튜터" | "분석 에이전트" |
| 색상 테마 | 기본 (주황 강조) | 파란 강조 (#1D6FDE) |
| 입력 placeholder | "궁금한 것을 물어보세요" | "분석하고 싶은 내용을 입력하세요" |

#### 방법 2 — 진입점 명확화

- 기본값: 현재 페이지에 따라 자동 선택 (Home → 분석, 교육 → 대화)
