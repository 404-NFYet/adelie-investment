# 손영진 (YJ99Son) — 프로젝트 변경이력

> 역할: 프론트엔드 개발 (팀장)
> 기간: 2026-01-20 ~ 현재 (개발 진행 중)
> 기술 스택: React 19, Vite, Tailwind CSS, ECharts, Plotly, Nginx

---

## Phase 1: 프로젝트 초기 구축 (01-20 ~ 02-06)

### Situation

Adelie Investment는 AI 기반 금융 교육 플랫폼으로, 현재 한국 주식 시장의 이벤트와 과거 역사적 사례를 매칭하여 "역사는 반복된다"는 관점에서 투자 교육을 제공하는 서비스이다. 프로젝트 시작 시점에서 프론트엔드 코드베이스는 완전히 비어 있었으며, Figma 디자인 시안과 PRD 문서만 존재하는 상태에서 전체 UI를 처음부터 구축해야 했다.

팀장으로서 기획 방향 수립과 기술 스택 선정을 동시에 진행해야 하는 상황이었다. 특히 모바일 퍼스트(max-width: 480px) 전략을 기반으로 한 SPA 아키텍처를 설계해야 했고, 백엔드 API 스키마가 병렬로 개발되는 환경에서 프론트엔드가 독립적으로 진행 가능한 구조를 만들어야 했다. 또한 DB 스키마 설계, PRD 작성, AI 파이프라인 기초 도구 함수 등 프론트엔드 외의 기반 작업에도 참여가 필요했다.

### Task

- React 19 + Vite + Tailwind CSS 기반 SPA 프론트엔드 프로젝트 초기화 및 아키텍처 설계
- PRD 문서 작성 및 DB 스키마(DBML) 설계
- 컴포넌트 디렉토리 구조 설계 및 공통 컴포넌트 구현
- 메인 레이아웃, 페이지 라우팅, 인증 플로우 기본 구현
- Playwright 기반 E2E 테스트 환경 구축
- AI 도구 함수(브리핑 생성, 기업분석, 용어집 로더) 작성

### Action

#### 프로젝트 기초 설계

- **React 19 + Vite 프론트엔드 초기화**: 최신 React 기능(Suspense, Lazy Loading)과 Vite의 빠른 HMR을 활용하기 위해 React 19 + Vite 조합을 선택하였다. Tailwind CSS를 스타일 프레임워크로 채택하여, CSS 변수 기반 테마 시스템과 다크모드(`class` 전략)를 초기부터 설계하였다. 브랜드 컬러(`#FF6B00`, 오렌지)를 CSS 변수로 관리하여 전체 UI의 시각적 일관성을 확보하였다.
- **PRD 및 DB 스키마 작성**: 프로젝트 요구사항 정의서(PRD)를 작성하고, dbdiagram.io용 DBML 포맷으로 데이터베이스 스키마를 설계하였다. 이를 통해 백엔드 팀원들과의 데이터 구조 합의를 이끌어냈다.
- **컴포넌트 디렉토리 구조 설계**: `common/`(재사용 범용), `domain/`(비즈니스 로직), `layout/`(AppHeader, BottomNav, ChatFAB), `charts/`(시각화), `tutor/`(AI 채팅 UI), `trading/`(모의투자) 등 역할별로 명확히 분리된 컴포넌트 계층 구조를 수립하였다.

#### 핵심 레이아웃 및 라우팅

- **메인 레이아웃 및 페이지 라우팅 구현**: React Router 기반의 SPA 라우팅을 구축하고, `App.jsx`에서 `React.lazy`를 활용한 페이지 단위 코드 스플리팅을 초기부터 적용하였다. Context 기반 전역 상태 관리 아키텍처(`ThemeProvider > UserProvider > PortfolioProvider > TutorProvider > TermProvider`)를 설계하여, 외부 상태 관리 라이브러리 없이도 효율적인 전역 상태 관리를 달성하였다.
- **API 클라이언트 모듈 구현**: `src/api/client.js`에 `fetchJson`/`postJson` 유틸리티를 구현하고, 도메인별 API 파일 분리 패턴을 확립하였다. 프로덕션에서는 빈 문자열(nginx 프록시), 로컬 개발에서는 `VITE_FASTAPI_URL` 환경변수를 사용하는 이중 전략을 채택하였다.
- **Playwright E2E 테스트 환경 세팅**: 인증 플로우를 포함한 E2E 테스트 기반을 구축하여, 프론트엔드의 품질 검증 체계를 초기부터 마련하였다.

#### AI 파이프라인 기초 도구

- **브리핑 생성 및 기업분석 AI 도구 함수**: 챗봇 서비스에서 사용할 브리핑 생성 도구와 기업 분석 도구 함수를 작성하여, AI 개발 팀원의 LangGraph 에이전트 통합을 지원하였다.
- **용어집 로더 리팩토링 및 키워드 추출 스크립트**: 금융 용어 사전 데이터를 효율적으로 로드하는 모듈을 리팩토링하고, 뉴스 데이터에서 키워드를 추출하는 스크립트를 추가하였다.

### Result

| 지표 | 수치 |
|------|------|
| 커밋 수 | 약 17개 |
| 주요 산출물 | PRD 문서, DBML 스키마, React 프로젝트 기본 구조, E2E 테스트 환경 |
| 구현 기반 | 6개 컴포넌트 카테고리, Context 5-layer 아키텍처, 도메인별 API 레이어 |

- React 19 + Vite + Tailwind CSS 기반의 확장 가능한 SPA 아키텍처를 수립하여, 이후 Phase에서 팀원들이 독립적으로 기능을 추가할 수 있는 견고한 기반을 마련하였다.
- PRD와 DB 스키마를 팀장으로서 주도적으로 작성하여, 백엔드/AI/인프라 팀원 간의 기술적 합의와 개발 방향성을 정립하였다.

## Phase 2: 핵심 기능 구현 (02-06 ~ 02-12)

### Situation

Phase 1에서 프론트엔드 기본 구조가 완성된 이후, 실제 사용자에게 제공할 핵심 페이지와 기능을 본격적으로 구현해야 하는 단계에 진입하였다. 백엔드 API(내러티브, 키워드, 포트폴리오, 튜터)가 병렬로 개발되면서 프론트엔드 연동 인터페이스를 빠르게 맞춰야 했고, Docker 기반 개발 환경에서의 프론트-백 통합 테스트가 빈번해졌다.

특히 7단계 내러티브 캐러셀, 차트 시각화, 모의투자, AI 튜터 채팅 등 기술적 난이도가 높은 기능들이 집중되어 있었다. 또한 PWA 지원, 알림 시스템, 리더보드, 피드백 시스템 등 사용자 경험을 풍부하게 하는 부가 기능도 이 시기에 구현이 필요했다. 팀장으로서 5명의 팀원 간 인터페이스 조율과 Git 전략 수립까지 병행해야 하는 상황이었다.

### Task

- 내러티브(브리핑) 캐러셀, 홈, 검색, 포트폴리오, 프로필 등 핵심 페이지 구현
- Plotly 기반 차트 시각화 컴포넌트 구현 및 모바일 최적화
- 모의투자(주식 시뮬레이션) 기능 구현 — 매수/매도 UI, 종목 검색, 미니 차트
- PWA 서비스 워커 + 피드백 시스템 구축
- AI 튜터 챗봇 UI 및 SSE 스트리밍 연동
- 알림 시스템, 리더보드, 체류 보상 훅 등 부가 기능 구현
- Docker 컨테이너화 및 LXD 환경 구축 지원

### Action

#### 핵심 페이지 구현

- **7단계 내러티브 캐러셀 통합**: 데이터 파이프라인이 생성하는 7단계(이후 6페이지로 전환) 내러티브 콘텐츠를 스와이프 기반 캐러셀로 구현하였다. 각 페이지별 차트/텍스트 레이아웃을 최적화하고, 모바일 환경에서의 터치 인터랙션을 고려한 네비게이션 UX를 설계하였다.
- **홈 키워드 카드 + 매칭 페이지**: 오늘의 키워드 카드 목록을 표시하는 홈 페이지와, 키워드 클릭 시 관련 기업 섹션과 모의투자 연동이 가능한 매칭 페이지를 구현하였다. 키워드 API의 stocks 데이터를 정규화하고 기업 선정 이유를 노출하는 UI를 추가하였다.
- **투자 탭 전면 구현**: 주식 검색, 매수/매도 모달, 포트폴리오 4탭 구조(보유종목/거래내역/수익/리더보드)를 구현하여 모의투자 기능을 완성하였다.

#### 차트 시각화 및 데이터 렌더링

- **Plotly 차트 컴포넌트 구현**: 서버에서 Plotly JSON 포맷으로 전달되는 차트 데이터를 렌더링하는 범용 차트 컴포넌트를 구현하였다. 챗봇 시각화 SSE 파싱과 iframe 렌더링의 안정성을 개선하고, 차트 CSS를 변수화하여 다크모드 대응을 완료하였다.
- **내러티브 UI 개선 및 mark 태그 렌더링**: 키워드 카드와 매칭 페이지에서 `<mark>` 태그 기반 하이라이팅 렌더링을 개선하고, 내러티브 차트 컴포넌트에 트래킹 기능을 추가하였다.

#### 사용자 경험 확장

- **PWA + 피드백 시스템 구축**: Service Worker를 도입하여 오프라인 캐싱과 앱 설치를 지원하고, 사용자 피드백 수집 위젯을 구현하였다. 스플래시 스크린과 온보딩 플로우를 설계하여 첫 사용자 경험을 강화하였다.
- **알림 시스템 + 체류 보상 + 리더보드**: 알림 페이지와 뱃지 표시 기능을 구현하고, 콘텐츠 체류 시간 기반의 포인트 보상 훅(`useDwellReward`)을 추가하였다. 리더보드 UI를 포트폴리오 탭 내에 통합하였다.
- **챗봇 출처 표기(Source Citation) 기능**: AI 튜터의 응답에 출처 정보를 시각적으로 표시하는 기능을 연결하여, 교육 콘텐츠의 신뢰성을 높였다.
- **온보딩 리디자인 + UX 전면 개선**: 하이라이팅, 로고 아이콘 최적화(흰 배경 제거 + favicon 정상화), 게스트 인증 유도, 로그아웃 플로우 등 전반적인 UX 품질을 개선하였다.

#### 기술적 의사결정

- **Git 전략 수립 및 모노레포 구조 정비**: 팀장으로서 `main ← develop ← feature/*` 브랜치 전략을 수립하고, git worktree 기반 병렬 개발 패턴을 팀에 도입하였다.
- **Docker 컨테이너화 + LXD 환경 구축**: 프론트엔드 Dockerfile 작성과 함께, 인프라 팀원과 협력하여 Docker Compose 기반의 개발 환경을 구축하였다.
- **데이터 파이프라인 기여**: Perplexity API 기반 뉴스 수집으로의 전환, RSS 완전 제거, Key Insight API 노출, 종목 선정 이론 반영 등 데이터 파이프라인의 방향성에도 팀장으로서 관여하였다.

### Result

| 지표 | 수치 |
|------|------|
| 커밋 수 | 약 30개 |
| 구현 페이지 수 | 8+ (홈, 내러티브, 검색, 포트폴리오, 매칭, 프로필, 알림, 온보딩) |
| 컴포넌트 구현 | 40+ (도메인, 레이아웃, 차트, 튜터, 트레이딩 포함) |
| 부가 기능 | PWA, 피드백 위젯, 알림 뱃지, 체류 보상, 리더보드, 챗봇 출처 표기 |

- 프론트엔드 핵심 페이지와 기능을 집중적으로 구현하여, 서비스의 전체 사용자 플로우를 완성하였다.
- 팀장으로서 Git 전략, 디렉토리 구조, 인터페이스 규약을 정의하여 5명의 팀원이 병렬로 효율적으로 개발할 수 있는 환경을 조성하였다.

## Phase 3: 통합 & 최적화 (02-12 ~ 02-14)

### Situation

Phase 2에서 구현된 프론트엔드, 백엔드, 데이터 파이프라인의 개별 기능들이 실제로 연동되는 E2E 통합 단계에 진입하였다. Spring Boot 인증 서버가 제거되고 FastAPI로 단일 백엔드가 통합되면서, 프론트엔드의 API 호출 경로와 nginx 프록시 설정을 전면 전환해야 했다. 또한 Service Worker의 캐싱 전략이 대용량 Plotly 번들(약 3MB)과 충돌하여 초기 로딩 성능 문제가 발생하였고, Context Provider의 불필요한 리렌더링으로 인한 성능 저하도 관측되었다.

포트폴리오 페이지에서 무한 로딩 상태에 빠지는 심각한 버그가 보고되었고, 모의투자 매매 UX에서도 사용자 혼란을 유발하는 이슈들이 다수 발견되었다. 전반적으로 개별 기능의 완성도를 높이고, 시스템 전체의 안정성을 확보하는 것이 이 시기의 핵심 과제였다.

### Task

- Spring Boot → FastAPI 인증 전환에 따른 프론트엔드 API 클라이언트 및 nginx 프록시 마이그레이션
- Service Worker 캐싱 전략 재설계 및 번들 최적화
- Context Provider 메모이제이션 및 리스트 컴포넌트 성능 최적화
- 포트폴리오 무한 로딩 버그 해결, 모의투자 매매 UX 개선
- URL 라우팅 패턴 정리 및 PWA 업데이트 알림 배너 추가
- 폼 필드 접근성(a11y) 개선

### Action

#### 백엔드 전환 대응

- **프론트엔드 auth API FastAPI 전환 + SPRING_URL 제거**: Spring Boot 인증 서버가 제거됨에 따라, `src/api/auth.js`의 API 호출 경로를 FastAPI 엔드포인트로 전면 전환하였다. `VITE_SPRING_URL` 환경변수를 제거하고, 모든 API가 단일 FastAPI 서버(`/api/v1/*`)를 통하도록 통합하였다.
- **nginx 프록시 Spring Boot → FastAPI 전환**: nginx 설정에서 Spring Boot 프록시 룰을 제거하고, `/api/v1/*`을 backend-api 컨테이너(`:8082`)로 프록시하는 단일 구성으로 전환하였다. 레거시 `/api/auth/*` 경로는 `/api/v1/auth/*`로 rewrite하여 하위 호환성을 유지하였다.

#### 성능 최적화

- **SW 캐싱 경량화 + 번들 최적화**: Service Worker의 precaching 대상에서 Plotly 대용량 청크(약 3MB)를 제외하고, HTML만 precache하도록 전략을 축소하였다. `autoUpdate` 모드를 적용하고, Vite의 `manualChunks` 설정으로 vendor 번들을 분리하여 캐시 효율을 극대화하였다. 사용되지 않던 React Query 의존성도 제거하여 번들 크기를 절감하였다.
- **Context Provider value 메모이제이션**: `PortfolioContext`, `TermContext`, `ThemeContext`, `TutorContext`, `UserContext` 총 5개 Context Provider에 `useCallback`과 `useMemo`를 적용하여, value 객체의 참조 안정성을 확보하고 불필요한 하위 트리 리렌더링을 방지하였다.
- **리스트 컴포넌트 React.memo 적용**: `KeywordCard`, `MessageBubble`, `HoldingCard`, `TradeItem` 등 반복 렌더링되는 리스트 아이템 컴포넌트에 `React.memo`를 적용하여 렌더링 성능을 개선하였다.

#### 버그 수정 및 UX 개선

- **포트폴리오 무한 로딩 4-layer fix**: 포트폴리오 페이지의 무한 로딩 문제를 백엔드 서비스(`stock_price_service.py` 타임아웃 처리), API 클라이언트(에러 바운더리 추가), PortfolioContext(로딩 상태 초기화), UserContext(인증 실패 시 복구), Portfolio 페이지(fallback UI) 등 4개 레이어에 걸쳐 종합적으로 해결하였다.
- **모의투자 매매 UX 개선**: TradeModal의 매수/매도 플로우를 전면 개선하였다. 주문 수량 입력 검증, 잔고 초과 방지, 매도 가능 수량 표시, 주문 확인 단계 등을 추가하여 사용자의 매매 실수를 줄였다.
- **회원가입 UI 개선 + AI 튜터 채팅 차단**: 회원가입 폼의 유효성 검증을 강화하고, 미인증 사용자의 AI 튜터 접근을 차단하는 가드 로직을 추가하였다.
- **URL 라우팅 패턴 정리**: `/case/:caseId`와 `/narrative/:caseId` 경로를 일관된 패턴으로 정리하고, 불필요한 라우팅 코드를 제거하였다.
- **PWA 업데이트 알림 배너**: Service Worker가 새 버전을 감지했을 때 사용자에게 업데이트 안내 배너를 표시하는 `UpdatePrompt` 컴포넌트를 구현하였다.
- **폼 필드 접근성 개선**: FeedbackWidget, TradeModal, StockSearch, ChatInput, TutorPanel, Auth, Profile, Search 등 8개 컴포넌트에 `id`/`name` 속성과 `<label>` 연결을 추가하여 스크린 리더 호환성을 확보하였다.

## Phase 4: 기능 고도화 (02-14 ~ 02-17)

### Situation

Phase 3에서 시스템 통합과 안정화가 완료된 이후, 서비스의 완성도를 높이기 위한 고도화 단계에 진입하였다. Figma 디자인 시안이 최종 확정되면서 랜딩 페이지를 포함한 UI 전면 재구현이 필요했고, 데이터 파이프라인이 생성하는 차트가 Plotly JSON 포맷인 반면 프론트엔드에서는 경량의 ECharts로 렌더링하는 것이 모바일 성능에 유리하다는 판단이 내려졌다.

내러티브 콘텐츠의 품질이 파이프라인 → API → 프론트엔드를 거치면서 저하되는 문제가 관측되었고, 차트 렌더링 실패 시의 폴백 전략이 부재하여 빈 화면이 노출되는 상황이 발생하고 있었다. 또한 마스코트 이미지(penguin-3d.png)가 8.2MB로 과도하게 커서 초기 로딩 속도에 심각한 영향을 미치고 있었다.

### Task

- Figma 디자인 기반 랜딩 페이지 5페이지 재구현 (스와이프 자동 전환 포함)
- Plotly → ECharts 변환 어댑터(`plotlyToEcharts`) 구현 및 차트 컴포넌트 이원화
- 내러티브 UI 전면 재구성 및 E2E 콘텐츠 품질 개선
- ChatFAB 활성화 및 페이지별 맞춤 프롬프트 연동
- 키워드 아이콘 매핑 파이프라인 도입 (LLM 기반 자동 아이콘 선택)
- 이미지 에셋 최적화 및 빌드 성능 개선

### Action

#### Figma 기반 UI 재구현

- **랜딩 페이지 Figma 컴포넌트 구조 재구현**: Figma 디자인 시안의 5페이지 랜딩을 React 컴포넌트로 1:1 재구현하였다. 각 페이지에 펭귄 마스코트 일러스트(analyst, book, magnifier, question)를 배치하고, 자동 전환 타이머와 스와이프 제스처를 지원하는 풀스크린 캐러셀을 구현하였다. 비로그인 사용자의 첫 진입점으로서 서비스 소개 플로우를 완성하였다.
- **랜딩-홈-콘텐츠 흐름 및 하단탭 UI 반영**: 사용자의 메인 네비게이션 경로를 `BottomNav` 컴포넌트와 함께 재설계하였다. 랜딩 → 홈 → 내러티브의 자연스러운 탐색 경험을 구현하고, 하단 탭 바의 아이콘과 라벨을 Figma 시안에 맞게 전면 교체하였다.

#### ECharts 마이그레이션 및 차트 안정화

- **plotlyToEcharts 변환 어댑터 구현**: 데이터 파이프라인이 Plotly JSON 포맷으로 생성하는 차트 데이터를 ECharts 옵션으로 변환하는 406줄 규모의 어댑터 유틸리티(`plotlyToEcharts.js`)를 개발하였다. line, bar, pie, scatter, area 등 주요 차트 타입의 변환을 지원하며, 날짜축/숫자축 자동 감지, 색상 매핑, 축 라벨/제목 변환, 반응형 크기 조절 등을 처리한다.
- **ResponsiveEChart + ResponsivePlotly 이원 컴포넌트**: ECharts 기반의 `ResponsiveEChart`와 Plotly 기반의 `ResponsivePlotly`를 모두 구현하여, 차트 데이터 형식에 따라 적절한 렌더러를 선택할 수 있는 유연한 구조를 마련하였다. 각 컴포넌트에 Error Boundary를 내장하여, 렌더링 실패 시에도 빈 화면 대신 안내 메시지를 표시하도록 하였다.
- **내러티브 차트 어댑터(narrativeChartAdapter.js)**: 내러티브 페이지 전용으로 파이프라인 출력의 차트 데이터를 정규화하는 어댑터를 별도로 구현하여, 다양한 차트 데이터 포맷에 대한 방어적 처리를 추가하였다.
- **종목 상세 바텀시트 라인차트 안정화**: 종목 상세 바텀시트의 주가 차트를 MiniChart(캔들스틱) 기반에서 ECharts 라인차트로 전환하고, 데이터 로딩 및 렌더링 안정성을 확보하였다.

#### 내러티브 콘텐츠 품질 개선

- **E2E 내러티브 콘텐츠 정제**: 파이프라인(`interface2`, `interface3`) → FastAPI(`narrative_builder`) → 프론트엔드(`Narrative.jsx`)를 관통하는 내러티브 콘텐츠의 표시 품질을 종단 간 개선하였다. 프롬프트 템플릿 6개를 수정하고, narrative_builder 서비스의 데이터 변환 로직을 개선하였으며, 프론트엔드의 렌더링 로직에서 엣지 케이스 처리를 추가하였다.
- **narrative spine 및 openai-first 생성 정책**: 내러티브 구조(spine)를 정의하고, OpenAI를 우선 프로바이더로 하는 생성 정책을 파이프라인과 프론트엔드 양측에 반영하였다. 테스트 코드(`test_interface3_story_spine.py`)를 작성하여 spine 구조의 일관성을 검증하였다.
- **차트 생성 재시도와 LLM 폴백 안정화**: chart_agent의 차트 데이터 생성 실패 시 최대 3회 재시도와 LLM 프로바이더 자동 폴백 메커니즘을 구현하였다. JSON 파싱 실패에 대한 retry 로직과 fallback 전략을 추가하여 파이프라인의 전체 성공률을 향상시켰다.

#### AI 튜터 및 키워드 아이콘

- **ChatFAB 활성화 + 페이지별 맞춤 프롬프트**: 전역 AI 튜터 채팅 버튼(ChatFAB)을 활성화하고, `TutorContext`를 통해 현재 페이지 컨텍스트에 맞는 프롬프트가 자동으로 설정되도록 구현하였다. 내러티브 페이지에서는 해당 브리핑 내용을, 홈 페이지에서는 오늘의 키워드 기반 질문을 컨텍스트로 전달한다.
- **키워드 아이콘 매핑 파이프라인 도입**: LLM이 키워드의 의미를 분석하여 20종의 Figma 아이콘 중 가장 적합한 아이콘을 자동 매핑하는 파이프라인 노드를 신규 구현하였다. `datapipeline/constants/home_icons.py`에 아이콘 카탈로그를 정의하고, `3_home_icon_map.md` 프롬프트 템플릿으로 LLM에 분류를 위임하는 구조를 설계하였다. 프론트엔드에서는 `homeIconCatalog.js`로 아이콘을 매핑하여 홈 키워드 카드에 시각적 아이덴티티를 부여하였다.

#### 에셋 최적화

- **penguin-3d 이미지 8.2MB → 330KB WebP 변환**: 마스코트 이미지를 PNG에서 WebP로 변환하여 96%의 용량을 절감하였다. `PenguinMascot`, `SplashScreen`, `Onboarding`, `FeedbackWidget`, `InstallPrompt`, `TutorModal` 등 6개 컴포넌트의 이미지 참조를 일괄 수정하였다. 동시에 nginx에 보안 헤더(X-Frame-Options, X-Content-Type-Options, CSP)를 추가하고, Vite 빌드 설정을 최적화하였다.
