Q4 프로덕션 리소스 제한은? (backend: 8G mem/4 CPU, postgres: 16G/4 CPU) +

프로덕션 환경에서는 Docker Compose의 deploy.resources로 서비스별 리소스 상한/하한을 설정합니다. deploy-test 서버(16CPU/32GB) 자원을 효율적으로 분배합니다.

서비스별 리소스 배분

서비스Memory LimitCPU LimitMemory ReservationCPU Reservation
postgres (pgvector:pg16)16G4--
backend-api (FastAPI)8G44G2
redis (7-alpine)2G2--
frontend (nginx)1G2--
docker-compose.prod.yml 
# PostgreSQL: 가장 큰 리소스 할당 (pgvector 벡터 연산 대응)
postgres:
  image: pgvector/pgvector:pg16
  deploy:
    resources:
      limits:
        memory: 16G
        cpus: "4"

# Backend API: reservation으로 최소 보장 + limit으로 상한
backend-api:
  deploy:
    resources:
      limits:
        memory: 8G
        cpus: "4"
      reservations:
        memory: 4G
        cpus: "2"

# Redis: appendonly + LRU 정책
redis:
  command: redis-server --appendonly yes --maxmemory 1gb --maxmemory-policy allkeys-lru
  deploy:
    resources:
      limits:
        memory: 2G
        cpus: "2"

로그 로테이션 (전 서비스 공통)

logging:
  driver: "json-file"
  options:
    max-size: "10m"   # 로그 파일당 최대 10MB
    max-file: "3"     # 최대 3개 파일 (총 30MB)

설계 의도: PostgreSQL에 가장 큰 리소스를 할당한 이유는 pgvector 확장의 벡터 유사도 검색(cosine similarity)이 메모리 집약적이기 때문입니다. backend-api는 reservation으로 최소 4G/2CPU를 보장하면서, LLM 호출이나 파이프라인 실행 시 8G/4CPU까지 확장됩니다.

Q5 CI/CD 워크플로우는? (ci.yml: lint+test, deploy.yml: build+push+deploy) +

GitHub Actions로 5개 워크플로우를 운영합니다. CI와 CD를 분리하여 PR에서는 검증만, develop push에서는 자동 배포가 진행됩니다.

CI 파이프라인 (ci.yml) - 4개 병렬 Job

.github/workflows/ci.yml 
on:
  pull_request:
    branches: [develop, main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true    # 같은 PR의 이전 실행 자동 취소

jobs:
  changes:        # 1. 경로 필터 (dorny/paths-filter)
  lint-python:    # 2. ruff check + format (Python 변경 시만)
  lint-frontend:  # 3. eslint (frontend 변경 시만)
  test-backend:   # 4. pytest (postgres + redis 서비스 컨테이너)
  build-check:    # 5. npm run build (프론트엔드 빌드 검증)

CD 파이프라인 (deploy.yml) - 단일 Job 5단계

.github/workflows/deploy.yml 
on:
  push:
    branches: [develop]

concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: false    # 배포는 취소하지 않음

jobs:
  deploy:
    runs-on: self-hosted       # LXD 호스트에서 실행
    steps:
      # 1단계: Build Docker images (3개)
      - name: Build frontend/backend-api/ai-pipeline

      # 2단계: Push to Docker Hub
      - name: Login & Push (docker/login-action)

      # 3단계: Deploy via SSH (appleboy/ssh-action)
      - name: SSH → git pull → docker compose up -d

      # 4단계: Health check (5회 재시도)
      - name: curl /api/v1/health + localhost:80

      # 5단계: Discord 알림 (성공/실패)
      - name: Discord notification

Q6 경로 필터(path-filter)로 선택적 CI는 어떻게 하나? +

dorny/paths-filter@v3 액션으로 변경된 파일 경로를 감지하고, 관련 CI Job만 선택적으로 실행합니다. 불필요한 빌드/테스트를 건너뛰어 CI 시간을 절약합니다.

경로 필터 정의

.github/workflows/ci.yml 
jobs:
  changes:
    runs-on: ubuntu-latest
    outputs:
      frontend: ${{ steps.filter.outputs.frontend }}
      python: ${{ steps.filter.outputs.python }}
    steps:
      - uses: actions/checkout@v4
      - id: filter
        uses: dorny/paths-filter@v3
        with:
          filters: |
            frontend:
              - "frontend/**"
            python:
              - "fastapi/**"
              - "datapipeline/**"
              - "chatbot/**"
              - "tests/**"

조건부 Job 실행

# Python 파일이 변경된 경우에만 lint + test 실행
lint-python:
  needs: changes
  if: needs.changes.outputs.python == 'true'
  runs-on: ubuntu-latest
  steps:
    - run: ruff check fastapi/ datapipeline/ chatbot/
    - run: ruff format --check fastapi/ datapipeline/ chatbot/

# Frontend 파일이 변경된 경우에만 lint + build check 실행
lint-frontend:
  needs: changes
  if: needs.changes.outputs.frontend == 'true'

build-check:
  needs: changes
  if: needs.changes.outputs.frontend == 'true'

Q8 Self-hosted Runner 선택 이유와 구성은? +

deploy.yml의 배포 워크플로우만 self-hosted runner를 사용하고, 나머지(CI, 알림)는 GitHub 제공 ubuntu-latest를 사용합니다.

Self-hosted vs GitHub-hosted 분리

워크플로우Runner이유
ci.yml (lint, test)ubuntu-latest표준 환경, 병렬 실행
deploy.yml (배포)self-hosted내부 네트워크(10.10.10.x) 접근 필요
main.yml (알림)ubuntu-latest외부 Webhook 호출만
claude.yml (AI)ubuntu-latestGitHub API 접근만

Self-hosted Runner를 사용하는 이유

  1. 내부 네트워크 접근: deploy-test(10.10.10.20)에 SSH로 직접 접근해야 하므로, LXD 호스트에서 실행되는 self-hosted runner가 필요합니다.
  2. Docker 이미지 캐시: 로컬에 Docker 이미지 레이어가 캐시되어 빌드 속도가 빠릅니다.
  3. 비용 절감: 대용량 Docker 이미지 빌드(3개)를 GitHub-hosted에서 실행하면 Actions 분수 소진이 큽니다.
.github/workflows/deploy.yml 
jobs:
  deploy:
    runs-on: self-hosted    # LXD 호스트 머신에서 실행
    steps:
      # Docker 빌드 (로컬 캐시 활용)
      - name: Build frontend image
        run: docker build -t dorae222/adelie-frontend:latest ./frontend

      # SSH로 내부 서버 접근 (10.10.10.20)
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}    # 10.10.10.20
          script: |
            cd ~/adelie-investment
            docker compose -f docker-compose.prod.yml up -d
Q9 deploy-test 서버 배포 프로세스는? (SSH → git pull → docker compose up) +

배포는 GitHub Actions(자동) 또는 lxd/Makefile(수동) 두 경로로 진행됩니다. 두 방식 모두 동일한 5단계 과정을 거칩니다.

배포 흐름 (5단계)

1. 로컬 Docker 이미지 빌드 (frontend, backend-api, ai-pipeline)
2. Docker Hub push (dorae222/adelie-*)
3. SSH → deploy-test 서버 접속
4. git pull → docker compose pull → docker compose up -d
5. nginx reload + 헬스체크

자동 배포 (GitHub Actions)

.github/workflows/deploy.yml 
- name: Deploy via SSH
  uses: appleboy/ssh-action@v1
  with:
    host: ${{ secrets.DEPLOY_HOST }}
    username: ${{ secrets.DEPLOY_USER }}

Q10 Dockerfile 멀티스테이지 빌드 전략은? +

Frontend는 2단계 멀티스테이지(Node builder → nginx runtime), Backend와 Pipeline은 단일 스테이지이지만 레이어 캐시 최적화를 적용합니다.

Frontend: Node builder → nginx (2단계)

frontend/Dockerfile 
# Stage 1: Node 빌드 (dev dependencies 포함)
FROM node:20-alpine AS builder
WORKDIR /app

# 의존성 설치 (package*.json만 먼저 복사 → 캐시 최적화)
COPY package*.json ./
RUN npm ci --silent

# 소스 복사 + 빌드 (빌드 시 주입하는 환경변수)
COPY . .
ARG VITE_FASTAPI_URL=""
ARG VITE_CLARITY_ID=""
ARG VITE_POSTHOG_KEY=""
ARG VITE_POSTHOG_HOST="https://analytics.adelie-invest.com"
RUN npm run build

# Stage 2: nginx 런타임 (빌드 산출물만 복사)
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

이미지 크기 차이: builder 스테이지(~1.2GB node_modules)를 버리고 dist/(~5MB)만 nginx:alpine(~40MB)에 복사하므로, 최종 이미지는 약 45MB로 축소됩니다.

Q12 WEB_CONCURRENCY=1 설정 이유는? +

프로덕션에서 Uvicorn 워커 수를 1로 제한하는 이유는 APScheduler 스케줄러의 중복 실행 방지입니다.

문제 상황

FastAPI 앱 시작 시 lifespan 핸들러에서 APScheduler를 시작합니다. Uvicorn이 멀티 워커(기본 4개)로 실행되면, 각 워커가 독립적으로 스케줄러를 생성하여 같은 파이프라인이 워커 수만큼 중복 실행됩니다.

fastapi/app/main.py 
@asynccontextmanager
async def lifespan(app: FastAPI):
    """Application lifespan handler."""
    redis_cache = await get_redis_cache()
    start_scheduler()    # ← 각 워커마다 호출됨!
    yield
    stop_scheduler()
    await close_redis_cache()
fastapi/app/core/scheduler.py 
def start_scheduler():
    """스케줄러 시작. 모닝(KST 09:00) + 데일리(KST 16:10), 월-금."""
    global _scheduler
    if _scheduler is not None:
        return    # 같은 프로세스 내 중복 방지 (워커 간은 불가)

    _scheduler = AsyncIOScheduler()

    # 모닝 파이프라인: KST 09:00 = UTC 00:00
    _scheduler.add_job(
        run_morning_pipeline,
        trigger=CronTrigger(hour=0, minute=0, day_of_week="mon,tue,wed,thu,fri"),
        id="morning_pipeline",
        misfire_grace_time=3600,
    )

    # 레거시 데일리 파이프라인: KST 16:10 = UTC 07:10
    _scheduler.add_job(
        run_daily_pipeline,
        trigger=CronTrigger(hour=7, minute=10, day_of_week="mon,tue,wed,thu,fri"),
        id="daily_pipeline",
        misfire_grace_time=3600,
    )

해결: WEB_CONCURRENCY=1

docker-compose.prod.yml 
backend-api:
  environment:
    WEB_CONCURRENCY: "1"     # 스케줄러 단일 실행 보장 (멀티 워커 → 중복 스케줄 방지)
fastapi/Dockerfile

트레이드오프

항목멀티 워커 (4)단일 워커 (1)
동시 요청 처리높음 (CPU 바운드 병렬)asyncio 이벤트 루프로 충분
스케줄러 안정성중복 실행 위험단일 실행 보장
파이프라인 비용LLM API 호출 n배정상 1회

왜 단일 워커로 충분한가: FastAPI는 async/await 기반이므로 I/O 바운드 작업(DB 쿼리, LLM API 호출)에서 단일 워커의 이벤트 루프만으로도 높은 동시성을 확보할 수 있습니다. CPU 바운드 작업이 거의 없는 이 서비스에서 멀티 워커의 이점은 미미합니다.

@router.get("/health")
async def health_check(db: AsyncSession = Depends(get_db)) -> dict:
    """Health check: API 상태 + DB 연결 상태 반환."""
    db_status = "healthy"
    db_error = None

    try:
        await db.execute(text("SELECT 1"))   # DB 실제 쿼리 검증
    except Exception as e:
        db_status = "unhealthy"
        db_error = str(e)

    return {
        "status": "healthy" if db_status == "healthy" else "degraded",
        "services": {
            "api": "healthy",
            "database": db_status,
        },
        "error": db_error,
    }

Q14 개인 로컬 DB 환경 전환 이유와 방법은? +

2026-02-24부터 공유 DB(infra-server) → 개인 로컬 DB로 전환했습니다. 각 LXD 개발 서버가 docker-compose.dev.yml로 자체 PostgreSQL을 운영합니다.

전환 이유

  1. 스키마 충돌 방지: 여러 개발자가 동시에 Alembic 마이그레이션을 실행하면 migration head 충돌 발생
  2. 데이터 격리: 테스트 데이터가 다른 개발자의 환경에 영향을 줌
  3. 독립적 개발: infra-server 장애 시 모든 개발자가 작업 중단되는 SPOF 제거
  4. 네트워크 지연 제거: 로컬 DB 접근으로 쿼리 응답 시간 개선
