Q2 정의된 알림 규칙 5가지는? +

Grafana Alerting의 provisioning 파일에 infrastructure 그룹으로 5개 알림 규칙이 정의되어 있습니다. 평가 간격은 1분입니다.

알림조건forseverity
HighCPUCPU 사용률 > 90%5mwarning
HighMemory메모리 사용률 > 85%5mwarning
DiskAlmostFull디스크 사용률 > 80%10mcritical
ContainerDownadelie-* 컨테이너 미감지1mcritical
ContainerRestart5분 내 재시작 발생1mwarning
infra/monitoring/grafana/provisioning/alerting/rules.yml 
apiVersion: 1
groups:
  - orgId: 1
    name: infrastructure
    interval: 1m
    folder: "infrastructure"
    rules:
      - alert: HighCPU
        title: "High CPU"
        expr: >
          100 - (avg by(instance)
            (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 90
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "{{ $labels.instance }} CPU > 90%"

      - alert: DiskAlmostFull
        title: "Disk Almost Full"
        expr: >
          100 - (node_filesystem_avail_bytes{mountpoint="/"}
            / node_filesystem_size_bytes{mountpoint="/"} * 100) > 80
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "{{ $labels.instance }} Disk > 80%"

      - alert: ContainerDown
        title: "Container Down"
        expr: absent(container_last_seen{name=~"adelie-.*"})
        for: 1m
        labels:
          severity: critical

      - alert: ContainerRestart
        title: "Container Restart"
        expr: increase(container_restart_count{name=~"adelie-.*"}[5m]) > 0
        for: 1m
        labels:
          severity: warning

HighCPU는 irate로 5분 간격 순간 CPU 사용률을 계산하고, ContainerDown은 absent() 함수로 adelie-* 패턴의 컨테이너가 아예 존재하지 않을 때 감지합니다.

Grafana 내장 Alerting을 사용하여 Discord webhook과 이메일 두 채널로 알림을 발송합니다. contact point와 notification policy가 provisioning 파일로 관리됩니다.

Contact Points

          message: |
            {{ if gt (len .Alerts.Firing) 0 }}
              FIRING ({{ len .Alerts.Firing }}건)
            {{ end }}
            {{ range .Alerts }}
            - **{{ .Labels.alertname }}**: {{ .Annotations.summary }}
            {{ end }}

Notification Policy (라우팅)

infra/monitoring/grafana/provisioning/alerting/notification_policies.yml 
apiVersion: 1
policies:
  - orgId: 1
    receiver: discord-webhook       # 기본 수신자
    group_by:
      - alertname
    group_wait: 30s
    group_interval: 5m
    repeat_interval: 4h
    routes:
      - receiver: discord-webhook   # critical → 10초 대기, 1시간 반복
        matchers:
          - severity=critical
        group_wait: 10s
        repeat_interval: 1h
      - receiver: default-email     # warning → 이메일, 12시간 반복
        matchers:
          - severity=warning
        repeat_interval: 12h

라우팅 전략: critical 알림은 Discord로 즉시(10초 대기) 전송되며 1시간마다 반복합니다. warning 알림은 이메일로 12시간 간격으로 발송됩니다. Discord webhook URL은 환경변수 GRAFANA_DISCORD_WEBHOOK_URL로 주입합니다.

Q5 prometheus_fastapi_instrumentator 구성은? +

prometheus-fastapi-instrumentator 라이브러리를 사용하여 FastAPI 앱의 HTTP 요청을 자동으로 계측합니다. /metrics 엔드포인트를 자동 노출합니다.

fastapi/app/main.py 
# Prometheus 메트릭 — /metrics 엔드포인트 자동 노출
try:
    from prometheus_fastapi_instrumentator import Instrumentator
    Instrumentator(
        should_group_status_codes=True,       # 2xx, 3xx 등으로 그룹화
        should_ignore_untemplated=True,       # 템플릿 없는 경로 무시
        should_instrument_requests_inprogress=True,  # 진행 중 요청 게이지
        excluded_handlers=[
            "/metrics", "/docs", "/redoc", "/openapi.json", "/"
        ],
    ).instrument(app).expose(app, endpoint="/metrics", include_in_schema=False)
except ImportError:
    logger.warning("prometheus-fastapi-instrumentator 미설치 — /metrics 비활성화")

주요 설정 옵션

  • should_group_status_codes=True: HTTP 상태 코드를 2xx/3xx/4xx/5xx로 그룹화하여 카디널리티 감소
  • should_ignore_untemplated=True: 라우터에 등록되지 않은 경로의 메트릭 제외
  • should_instrument_requests_inprogress=True: 현재 처리 중인 요청 수를 Gauge로 추적
  • excluded_handlers: 문서, 메트릭, 루트 등 유틸리티 엔드포인트 제외
  • include_in_schema=False: OpenAPI 스키마에서 /metrics 숨김

try/except로 감싸서 라이브러리 미설치 시에도 애플리케이션이 정상 기동되도록 graceful하게 처리합니다.

자동 생성되는 메트릭

  • http_request_duration_seconds (Histogram) — 요청 처리 시간
  • http_requests_inprogress (Gauge) — 현재 처리 중 요청 수
  • http_request_size_bytes (Histogram) — 요청 크기
  • http_response_size_bytes (Histogram) — 응답 크기
Q6 DB 쿼리 메트릭은 어떻게 수집하나? +

SQLAlchemy의 이벤트 리스너를 활용하여 모든 DB 쿼리를 자동으로 계측합니다. 성공/실패 각각 별도의 이벤트 훅에서 Prometheus Counter를 증가시킵니다.

fastapi/app/core/database.py 
from sqlalchemy import event
from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine
from app.metrics import DB_QUERY_TOTAL

engine = create_async_engine(
    settings.DATABASE_URL,
    echo=settings.DEBUG,
    pool_pre_ping=True,
    pool_size=50,
    max_overflow=100,
    pool_timeout=30,
    pool_recycle=1800,
)


def _extract_db_operation(statement: str) -> str:
    """SQL 문의 첫 키워드를 추출하여 operation 라벨로 사용."""
    if not statement:
        return "other"
    op = statement.lstrip().split(None, 1)[0].lower()
    if op in {"select", "insert", "update", "delete"}:
        return op
    return "other"


@event.listens_for(engine.sync_engine, "after_cursor_execute")
def _after_cursor_execute(conn, cursor, statement, parameters, context, executemany):
    DB_QUERY_TOTAL.labels(_extract_db_operation(statement), "success").inc()


@event.listens_for(engine.sync_engine, "handle_error")
def _handle_db_error(exception_context):
    statement = getattr(exception_context, "statement", "") or ""
    DB_QUERY_TOTAL.labels(_extract_db_operation(statement), "fail").inc()

수집 흐름

  1. SQLAlchemy sync_engine의 after_cursor_execute 이벤트에 리스너 등록
  2. 쿼리 실행 후 SQL 문 첫 단어를 파싱하여 operation 라벨 결정 (select/insert/update/delete/other)
  3. 성공 시 DB_QUERY_TOTAL.labels(operation, "success").inc()
  4. 에러 시 handle_error 이벤트에서 DB_QUERY_TOTAL.labels(operation, "fail").inc()

이를 통해 Grafana에서 rate(db_query_total[1m])로 초당 쿼리 수를, db_query_total{result="fail"}로 에러율을 실시간 확인할 수 있습니다.

Q9 Prometheus 데이터 보존 기간은? +

Prometheus TSDB 보존 기간은 환경별로 다르게 설정됩니다.

환경보존 기간설정 위치
프로덕션 (deploy-test)30일docker-compose.yml
개발 (dev)기본값 (15일)prometheus.dev.yml (별도 지정 없음)

프로덕션 설정

infra/monitoring/docker-compose.yml (prometheus 서비스) 
  prometheus:
    image: prom/prometheus:latest
    command:
      - --config.file=/etc/prometheus/prometheus.yml
      - --storage.tsdb.retention.time=30d    # 30일 보존
      - --web.enable-lifecycle               # /-/reload API 활성화

--web.enable-lifecycle 플래그로 POST /-/reload API가 활성화되어 있어, 설정 파일 변경 후 재시작 없이 curl -X POST http://localhost:9090/-/reload로 설정을 리로드할 수 있습니다.

데이터 영속성은 Docker named volume prometheus_data에 저장되므로 컨테이너 재시작 시에도 유지됩니다.

Q10 LangSmith로 LLM 호출 추적은 어떻게 하나? +

LangSmith를 사용하여 데이터 파이프라인의 모든 LLM 호출을 추적합니다. shared/langsmith_config.py에 공통 설정이 있고, 각 모듈에서 @traceable 데코레이터로 개별 함수를 계측합니다.

공통 설정 (shared)

shared/langsmith_config.py 
import os
from langsmith import Client
from langsmith.run_helpers import traceable

def setup_langsmith() -> bool:
    """LangSmith 환경 변수 설정."""
    settings = get_settings()
    if not settings.langsmith.tracing_enabled:
        os.environ["LANGCHAIN_TRACING_V2"] = "false"
        return False

    # 환경에 따른 프로젝트명 설정
    project_name = f"{settings.langsmith.project_name}-{settings.environment}"

    os.environ["LANGCHAIN_TRACING_V2"] = "true"
    os.environ["LANGCHAIN_API_KEY"] = settings.langsmith.api_key
    os.environ["LANGCHAIN_PROJECT"] = project_name
    os.environ["LANGCHAIN_ENDPOINT"] = settings.langsmith.endpoint
    return True
