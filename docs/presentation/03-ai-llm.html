Q2MultiProviderClient 동시성 제어는? (전역 세마포어 6, 프로바이더별 4/2/2)+

MultiProviderClient는 2계층 세마포어 시스템으로 동시성을 제어합니다. 전역 세마포어가 전체 LLM 호출 수를 제한하고, 프로바이더별 세마포어가 개별 API의 Rate Limit를 존중합니다.

세마포어기본값환경변수목적

OpenAI4OPENAI_MAX_CONCURRENCYOpenAI API Rate Limit 보호

호출 시 전역 세마포어를 먼저 획득한 후, 프로바이더별 세마포어를 추가로 획득하는 이중 잠금 구조입니다. Context Manager 패턴으로 예외 발생 시에도 세마포어가 안전하게 해제됩니다.

# 2계층 세마포어 초기화
self._global_semaphore = threading.BoundedSemaphore(LLM_MAX_CONCURRENCY)
self._provider_semaphores: dict[str, threading.BoundedSemaphore] = {
    "openai": threading.BoundedSemaphore(OPENAI_MAX_CONCURRENCY),
    "perplexity": threading.BoundedSemaphore(PERPLEXITY_MAX_CONCURRENCY),
    "anthropic": threading.BoundedSemaphore(ANTHROPIC_MAX_CONCURRENCY),
}

@contextmanager
def _provider_slot(self, provider: str):
    """전역 + 프로바이더별 세마포어 이중 획득/해제"""
    provider_sem = self._provider_semaphores.get(provider)
    self._global_semaphore.acquire()
    if provider_sem is not None:
        provider_sem.acquire()
    try:
        yield
    finally:
        if provider_sem is not None:
            provider_sem.release()
        self._global_semaphore.release()
Q3재시도(retry) 로직은? (retryable 에러 키워드, 지수 백오프, 최대 1회)+

재시도 로직은 2개 레이어에서 동작합니다. MultiProviderClient의 내장 재시도와 llm_utils.py의 프로바이더 폴백입니다.

1단계: MultiProviderClient 내장 재시도

  • 최대 재시도 횟수: 기본 1회 (PROVIDER_MAX_RETRIES=1, 총 2회 시도)
  • 지수 백오프: base=1.0초, cap=8.0초, base * 2^(attempt-1) + jitter
  • 재시도 가능 에러 판별: 에러 메시지에 특정 키워드가 포함되면 재시도

Retryable 에러 키워드 목록

timeout, timed out, connection reset, connection aborted, temporarily unavailable, service unavailable, rate limit, too many requests, 429, 502, 503, 504, bad gateway

2단계: llm_utils 프로바이더 폴백

  • OpenAI 재시도: OpenAI 호출이 retryable 에러로 실패하면 동일 모델로 1회 재시도
  • Anthropic → OpenAI 폴백: Claude가 크레딧 부족, 미초기화, 쿼터 초과 등으로 실패하면 gpt-5.2로 자동 폴백

_RETRYABLE_ERROR_KEYWORDS = (
    "timeout", "timed out", "connection reset", "connection aborted",
    "temporarily unavailable", "service unavailable",
    "rate limit", "too many requests", "429", "502", "503", "504", "bad gateway",
)

def _is_retryable_error(self, exc: Exception) -> bool:
    if isinstance(exc, (TimeoutError, ConnectionError)):
        return True
    lower = str(exc).lower()
    return any(keyword in lower for keyword in _RETRYABLE_ERROR_KEYWORDS)

def _compute_backoff_seconds(self, attempt: int) -> float:
    base = max(0.1, PROVIDER_BACKOFF_BASE_SECONDS)
    cap = max(base, PROVIDER_BACKOFF_MAX_SECONDS)
    wait = min(cap, base * (2 ** max(0, attempt - 1)))
    return wait + random.uniform(0, min(0.5, wait * 0.2))  # jitter
Q4GPT-5 특수 처리는 무엇인가? (max_completion_tokens, reasoning_effort)+

GPT-5 계열 모델은 OpenAI의 다른 모델과 API 파라미터 체계가 다릅니다. MultiProviderClient는 모델명에 gpt-5가 포함되면 자동으로 특수 처리를 적용합니다.

GPT-5 vs 일반 모델 파라미터 차이

파라미터일반 모델 (gpt-4o-mini 등)GPT-5 계열
토큰 상한max_tokensmax_completion_tokens
temperaturefrontmatter에서 설정전달하지 않음 (GPT-5가 자체 관리)
thinking 모드미지원reasoning_effort (low/medium/high)

thinking 파라미터와 thinking_effort는 프롬프트 frontmatter에서 선언할 수 있으며, GPT-5 모델일 때만 reasoning_effort로 변환되어 전달됩니다. 현재 프로젝트에서는 3_home_icon_map.md가 gpt-5-mini를 사용하고 있습니다.

datapipeline/ai/multi_provider_client.py:196-225 
def _call_openai_compatible(self, provider, model, messages, thinking,
                            thinking_effort, temperature, max_tokens,
                            response_format, **kwargs):
    """OpenAI 호환 API 호출 (OpenAI, Perplexity)."""
    client = self.providers[provider]
    is_gpt5 = provider == "openai" and "gpt-5" in model

    call_kwargs = {"model": model, "messages": messages}

    if is_gpt5:
        # GPT-5: max_completion_tokens 사용, temperature 전달 안 함
        call_kwargs["max_completion_tokens"] = max_tokens
    else:
        # 일반 모델: max_tokens + temperature
        call_kwargs["max_tokens"] = max_tokens
        call_kwargs["temperature"] = temperature

    # thinking 모드: GPT-5에서만 reasoning_effort 전달
    if thinking and is_gpt5:
        call_kwargs["reasoning_effort"] = thinking_effort

    if response_format:
        call_kwargs["response_format"] = response_format

    response = client.chat.completions.create(**call_kwargs)
Q5프롬프트 템플릿 관리 방법은? (마크다운 + frontmatter)+

프롬프트 템플릿은 마크다운 파일(.md)에 YAML-like frontmatter를 붙인 형식으로 관리됩니다. prompt_loader.py가 이를 파싱하여 PromptSpec 데이터클래스를 반환합니다.

핵심 기능 3가지

  1. Include 지시자: {{include:_tone_guide}}로 공통 모듈(톤 가이드, 차트 스켈레톤)을 인라인 삽입. _ 접두어 파일은 include 전용 파셜입니다.
  2. 변수 치환: {{variable}} 플레이스홀더를 call_llm_with_prompt() 호출 시 넘긴 dict/list 값으로 치환. dict/list는 자동으로 JSON 직렬화됩니다.
  3. 메타데이터 분리: frontmatter에서 프로바이더, 모델, temperature 등을 선언하면 코드 수정 없이 모델 교체가 가능합니다.

Q7할루시네이션 방지를 위한 다단계 검증은?+

Adelie Investment는 최소 3단계, 최대 5단계의 할루시네이션 검증을 파이프라인에 내장하고 있습니다. 모든 검증은 curated_context(실제 수집 데이터)를 Ground Truth로 사용합니다.

검증 파이프라인 흐름

  1. Interface 2 팩트체크 (hallucination_check): page_purpose, historical_case, narrative 전체를 curated_context 기준으로 교차 검증. 수치/날짜/고유명사/인과관계를 검사하고 validated_interface_2를 반환합니다.
  2. Interface 3 페이지 팩트체크 (3_hallcheck_pages): theme, one_liner, 6개 pages를 validated_interface_2 기준으로 재검증. cross_page_consistency, timeline_consistency, numeric_consistency 체크를 수행합니다.
  3. Interface 3 용어 팩트체크 (3_hallcheck_glossary): 용어 정의를 웹 검색 결과(Search Results)와 대조하여 정의 정확성, 눈높이, 중복 여부를 검증합니다.
  4. 차트 팩트체크 (3_hallcheck_chart): 생성된 차트의 x/y 데이터가 원본 소스와 일치하는지, 시각적 오해 소지가 없는지 검증합니다.
  5. 최종 할루시네이션 체크 (final_hallucination): 6페이지 전체 브리핑을 curated_context 기준으로 종합 팩트체크합니다.

Q8LLM 비용 최적화 전략은?+

프로젝트는 여러 레이어에서 LLM 비용을 최적화합니다.

1. 모델 티어링 (Model Tiering)

작업 복잡도에 따라 모델을 분리합니다. 핵심 내러티브/팩트체크에는 Claude Sonnet 4.6을 사용하고, 구조화된 데이터(차트, 용어사전)에는 비용이 낮은 gpt-4o-mini를, 경량 작업(아이콘 매핑)에는 gpt-5-mini를 사용합니다.

2. 응답 캐시 (LLM Response Cache)

동일한 프롬프트+모델+메시지 조합의 응답을 SHA-256 해시 키로 인메모리 캐싱합니다. TTL 15분, 최대 512개 엔트리, LRU 방식으로 운영되어 파이프라인 실행 중 중복 호출을 제거합니다.

3. JSON 복구로 재호출 절약

LLM 응답의 JSON 파싱 실패 시, 로컬 복구(API 호출 없음)를 먼저 시도합니다. 코드블록 제거, 중괄호 균형 맞추기, 스마트 따옴표 변환, trailing comma 제거 등을 수행하여 불필요한 재호출을 방지합니다.

def _call_anthropic(self, model, messages, temperature, max_tokens):
    """Anthropic Claude API 호출."""
    # system 메시지 분리 (Anthropic은 별도 파라미터)
    system_msg = ""
    user_messages = []
    for msg in messages:
        if msg["role"] == "system":
            system_msg += msg["content"] + "\n"
        else:
            user_messages.append(msg)

    # user 메시지가 없으면 기본 메시지 추가
    if not user_messages:
        user_messages = [{"role": "user", "content": "위 지시사항을 수행해주세요."}]

    call_kwargs = {"model": model, "max_tokens": max_tokens,
                   "temperature": temperature, "messages": user_messages}
    if system_msg.strip():
        call_kwargs["system"] = system_msg.strip()

    response = client.messages.create(**call_kwargs)

    # content 블록 순회 (Anthropic 특유의 블록 구조)
    content = ""
    for block in response.content:
        if hasattr(block, "text"):
            content += block.text

    # 통일 형식으로 변환 (input_tokens → prompt_tokens)
    return {
        "choices": [{"message": {"content": content, "role": "assistant"}}],
        "model": response.model,
        "usage": {
            "prompt_tokens": getattr(response.usage, 'input_tokens', 0),
            "completion_tokens": getattr(response.usage, 'output_tokens', 0),
        },
    }
Q12프로바이더 타임아웃 설정은?+

각 프로바이더는 서로 다른 타임아웃을 설정합니다. 타임아웃은 네트워크 레벨에서 SDK에 의해 강제되며, 환경변수로 오버라이드할 수 있습니다.

프로바이더별 타임아웃 설정

프로바이더기본 타임아웃환경변수설정 위치
OpenAI180초 (3분)OPENAI_TIMEOUT_SECONDSOpenAI SDK timeout 파라미터
Perplexity60초 (1분)PERPLEXITY_TIMEOUT_SECONDSOpenAI SDK timeout 파라미터 (호환 API)
AnthropicSDK 기본값—Anthropic SDK 내부 기본값 사용

타임아웃 차이의 이유:

  • OpenAI (180초): 내러티브 생성, 차트 생성 등 복잡한 JSON 응답을 생성하는 작업이 있어 넉넉하게 설정
  • Perplexity (60초): 검색 기반 API로 비교적 빠른 응답이 기대되므로 짧게 설정
  • Anthropic (SDK 기본): Anthropic SDK가 자체 타임아웃을 관리하므로 별도 설정하지 않음

타임아웃 발생 시 _is_retryable_error()에서 timeout, timed out 키워드로 감지되어 자동 재시도 대상이 됩니다.
