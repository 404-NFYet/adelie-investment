Q1React 19 + Vite 6 선택 이유는?+

React 19 선택 이유

  • 자동 배칭(Automatic Batching) 강화 — 이벤트 핸들러뿐 아니라 Promise, setTimeout, native 이벤트에서도 상태 업데이트가 자동 배칭되어 렌더링 횟수가 감소합니다. SSE 스트리밍으로 실시간 메시지를 받는 튜터 채팅에서 빈번한 setMessages 호출이 자연스럽게 최적화됩니다.
  • useTransition / Suspense 개선 — 17개 페이지를 React.lazy로 코드 스플리팅하고, Suspense의 fallback(PenguinLoading)으로 로딩 UX를 관리합니다.
  • use() Hook 지원 — 비동기 리소스를 선언적으로 처리할 수 있어, Context 6단계 래핑에서의 데이터 전달이 깔끔합니다.
  • 서버 컴포넌트 호환 — 향후 SSR/RSC로의 점진적 전환 가능성을 열어둡니다.

Vite 6 선택 이유

  • 네이티브 ESM 기반 HMR — 번들링 없이 모듈 단위로 교체하여 개발 서버 시작과 핫 리로드가 밀리초 단위입니다.
  • Rollup 4 기반 빌드 — manualChunks로 plotly(1.7MB+), framer-motion, chart.js, analytics를 별도 청크로 분리하여 초기 로드를 최적화합니다.
  • PWA 플러그인 통합 — vite-plugin-pwa로 Workbox 기반 서비스 워커, 매니페스트, 캐싱 전략을 하나의 설정으로 관리합니다.
  • Proxy 설정의 간결함 — server.proxy로 개발 시 CORS 없이 백엔드 API를 호출합니다.

Q2Context Provider 구조 (7단계 래핑)는?+

앱 전체를 감싸는 Provider 래핑 순서는 의존성 방향에 따라 결정됩니다. 상위 Provider가 제공하는 값을 하위 Provider가 사용하므로, 순서가 바뀌면 런타임 에러가 발생합니다.

Q6API 클라이언트 구현은? (authFetch, 401 refresh retry, CSRF)+

중앙 집중식 HTTP 클라이언트(client.js)가 인증, CSRF, 에러 핸들링을 모두 처리합니다. 도메인별 API 모듈(auth.js, portfolio.js, narrative.js 등)은 이 클라이언트를 호출하기만 합니다.

핵심 흐름: authFetch

frontend/src/api/client.js:82-108 
/** 인증 포함 fetch (HttpOnly 쿠키 자동 전송, 401 시 refresh 후 1회 재시도) */
export const authFetch = async (url, options = {}, canRetry = true) => {
  const response = await fetch(url, {
    ...options,
    credentials: 'include',  // HttpOnly 쿠키 자동 전송
    headers: mergeHeaders(options.headers, options.method),
  });

  if (response.status !== 401 || !canRetry || isAuthBypassRequest(url)) {
    return response;
  }

  // 401 → refresh token으로 access token 갱신 후 1회 재시도
  try {
    await refreshAccessToken();
    const retried = await fetch(url, {
      ...options,
      credentials: 'include',
      headers: mergeHeaders(options.headers, options.method),
    });
    if (retried.status === 401) {
      handleAuthFailure();  // 재시도도 실패 → 강제 로그아웃
    }
    return retried;
  } catch (error) {
    handleAuthFailure();
    throw error;
  }
};

CSRF Double-Submit Cookie 패턴

POST/PUT/PATCH/DELETE 요청 시 쿠키에서 csrfToken을 읽어 X-CSRF-Token 헤더에 자동 포함합니다.

frontend/src/api/client.js:32-45 
const getCsrfToken = () => {
  const match = document.cookie.match(/(?:^|;\s*)csrfToken=([^;]*)/);
  return match ? decodeURIComponent(match[1]) : '';
};

const mergeHeaders = (headers, method) => {
  const merged = new Headers(headers || {});
  if (!['GET', 'HEAD', 'OPTIONS'].includes(method?.toUpperCase())) {
    const csrf = getCsrfToken();
    if (csrf) merged.set('X-CSRF-Token', csrf);
  }
  return merged;
};

Refresh Token 동시 요청 처리

refreshPromise 싱글턴으로 동시에 여러 401이 발생해도 refresh 요청은 1회만 실행됩니다.

frontend/src/api/client.js:61-79 
let refreshPromise = null;

const refreshAccessToken = async () => {
  if (refreshPromise) return refreshPromise;  // 이미 진행 중이면 재사용
  refreshPromise = (async () => {
    const response = await fetch(`${API_BASE_URL}/api/v1/auth/refresh`, {
      method: 'POST',
      credentials: 'include',
      headers: { 'Content-Type': 'application/json' },
    });
    if (!response.ok) {
      const message = await readErrorMessage(response);
      throw new Error(message);
    }
  })().finally(() => { refreshPromise = null; });
  return refreshPromise;
};

래퍼 함수

fetchJson(GET), postJson(POST), deleteJson(DELETE) 세 래퍼가 있으며 모두 authFetch를 내부에서 호출합니다.

강제 로그아웃

refresh 실패 시 window.dispatchEvent(new Event('auth:logout'))를 발행하여 UserContext가 수신하고 setUser(null)로 상태를 초기화합니다.

Q7SSE 클라이언트 구현은? (ReadableStream, 이벤트 타입 파싱)+

AI 튜터 채팅은 Server-Sent Events(SSE)를 fetch + ReadableStream으로 직접 구현합니다. EventSource API 대신 fetch를 사용하는 이유는 POST 요청에 JSON body를 보내야 하고, HttpOnly 쿠키 인증(credentials: 'include')이 필요하기 때문입니다.

성능 최적화: 120ms 플러시 간격

매 청크마다 setMessages를 호출하면 과도한 리렌더링이 발생합니다. STREAM_FLUSH_INTERVAL_MS = 120으로 설정하여 최소 120ms 간격으로만 UI를 갱신합니다.

const STREAM_FLUSH_INTERVAL_MS = 120;
let lastFlushTime = Date.now();
let pendingFlush = false;

// 청크 수신 후
if (pendingFlush && Date.now() - lastFlushTime >= STREAM_FLUSH_INTERVAL_MS) {
  setMessages((prev) =>
    prev.map((m) => (m.id === assistantMsgId ? { ...m, content: fullContent } : m))
  );
  lastFlushTime = Date.now();
  pendingFlush = false;
}

function ProtectedRoute({ children }) {
  const { user, isLoading } = useUser();
  if (isLoading) return <PageLoader />;       // 인증 확인 중 로딩
  if (!user?.isAuthenticated) return <Navigate to="/" replace />;  // 미인증 → 랜딩
  return children;
}

function CaseRedirect() {

}

인증 흐름 요약

  1. 앱 시작 → UserProvider가 authApi.getMe()로 쿠키 기반 인증 복원
  2. isLoading 동안 PageLoader(펭귄 애니메이션) 표시
  3. 인증 성공 → ProtectedRoute 통과, 인증 실패 → Landing(/)으로 리다이렉트
  4. API 호출 중 401 → client.js가 refresh 시도 → 실패 시 auth:logout 이벤트 → UserProvider가 상태 초기화
