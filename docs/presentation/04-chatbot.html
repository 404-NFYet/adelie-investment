Q1LangGraph 기반 튜터 에이전트 구조는? (AgentState, agent→tools 루프)+

튜터 에이전트는 LangGraph의 StateGraph를 사용하여 agent → tools → agent 순환 루프로 구현되어 있습니다. 상태(State)는 AgentState TypedDict로 정의되며, 메시지 이력, 난이도, 컨텍스트 타입/ID를 보관합니다.

chatbot/agent/tutor_agent.py 
class AgentState(TypedDict):
    """State for the agent graph."""
    messages: Annotated[Sequence, add_messages]
    difficulty: str
    context_type: Optional[str]
    context_id: Optional[int]

그래프 구성은 _build_graph() 메서드에서 수행됩니다. 두 개의 노드(agent, tools)를 정의하고, _should_continue 조건 함수가 LLM 응답에 tool_calls가 있는지 검사하여 "continue"(tools 노드로 이동) 또는 "end"(종료)를 결정합니다.

chatbot/agent/tutor_agent.py 
def _build_graph(self) -> StateGraph:
    graph = StateGraph(AgentState)

    # 노드 정의
    graph.add_node("agent", self._agent_node)
    graph.add_node("tools", ToolNode(self.tools))

    # 엣지 정의
    graph.add_edge(START, "agent")
    graph.add_conditional_edges(
        "agent",
        self._should_continue,
        {"continue": "tools", "end": END}
    )
    graph.add_edge("tools", "agent")  # 도구 실행 후 다시 agent로

    if self.checkpointer:
        return graph.compile(checkpointer=self.checkpointer)
    return graph.compile()

실행 흐름

  1. START → agent: 시스템 프롬프트 + 사용자 메시지를 LLM에 전달
  2. agent → _should_continue: 응답에 tool_calls가 있으면 "continue", 없으면 "end"
  3. tools: LangChain ToolNode가 도구를 실행하고 결과를 메시지에 추가
  4. tools → agent: 도구 결과를 포함한 메시지로 다시 LLM 호출
  5. 도구 호출이 없을 때까지 2~4를 반복하고 최종 텍스트 응답을 반환

LLM은 ChatOpenAI(model="gpt-4o-mini", temperature=0.7, streaming=True)로 초기화되며, bind_tools()를 통해 5개의 LangChain 도구가 바인딩됩니다. 세션 영속성이 필요한 경우 checkpointer를 그래프 컴파일 시 전달할 수 있습니다.

Q2Chart-First 아키텍처란? (차트 선생성 → 텍스트 스트리밍, 5단계 파이프라인)+

Chart-First 아키텍처는 시각화가 필요한 질문에 대해 텍스트 응답보다 차트를 먼저 생성하여 SSE로 전송하는 전략입니다. 사용자가 차트를 바로 보면서 후속 텍스트 설명을 읽을 수 있어 체감 응답 속도가 빨라집니다.

5단계 파이프라인

  1. 시각화 의도 판단: should_auto_visualize()로 사용자 요청 + 종목 감지 결과를 기반으로 차트 필요 여부를 결정합니다.
  2. 차트 유형 분류 (Step 1): classify_chart_request()가 LLM(gpt-4o-mini, temperature=0)을 호출하여 11가지 지원 차트 유형 중 하나를 JSON 형식으로 분류합니다.
  3. 차트 JSON 생성 (Step 2): generate_chart_json()이 실제 주가 데이터와 함께 Plotly JSON 구성을 생성합니다.
  4. 차트 SSE 전송: 생성된 Plotly JSON을 event: visualization으로 즉시 스트리밍합니다.
  5. 텍스트 스트리밍: 차트 데이터 요약을 시스템 프롬프트에 주입하고, LLM이 "차트를 기반으로 분석"하는 텍스트를 event: text_delta로 스트리밍합니다.

Q3Hybrid Clarification Flow란? (휴리스틱 → LLM 분류 → 선택지 제시)+

Hybrid Clarification Flow는 사용자의 모호한 질문을 감지하고 확인 질문을 제시하는 3단계 파이프라인입니다. 비용이 낮은 휴리스틱 필터를 먼저 적용한 뒤, LLM 분류기로 정밀 판단하고, 최종적으로 선택지 버튼을 SSE로 전송하는 구조입니다.

1단계: 휴리스틱 모호성 탐지

chatbot/services/tutor_orchestrator_graph.py 
def _heuristic_ambiguity(message: str) -> tuple[bool, list[str], str]:
    cleaned = message.strip()
    if not cleaned:
        return True, ["topic"], "입력이 비어 있음"

    pronoun_like = ("이거", "저거", "그거", "이 내용", "이 화면", "방금", "그냥")
    vague_verbs = ("해줘", "알려줘", "보여줘", "설명해줘", "자세히")
    has_pronoun = any(token in cleaned for token in pronoun_like)
    has_vague_verb = any(token in cleaned for token in vague_verbs)

    if len(cleaned) <= 8 and (has_pronoun or has_vague_verb):
        return True, ["topic"], "짧고 지시 대상이 불명확함"
    if has_pronoun and has_vague_verb:
        return True, ["topic"], "지시 대상/범위가 불명확함"
    return False, [], "명확한 요청"

2단계: LLM 분류기

휴리스틱 결과와 별개로, analyze_ambiguity() 노드에서 gpt-4o-mini를 temperature=0으로 호출하여 {"is_ambiguous": true, "missing_slots": ["topic", "period"], "reasoning": "..."} 형태의 JSON을 받습니다. API 장애 시 휴리스틱 결과로 폴백합니다.

3단계: 선택지 생성 및 SSE 전송

모호하다고 판정되면 build_clarification() 노드가 missing_slots에 따라 기간 선택지(PERIOD_OPTIONS) 또는 범위 선택지(GENERAL_OPTIONS)를 기본 제공하고, LLM을 호출하여 더 적절한 선택지를 생성합니다.

chatbot/services/tutor_engine.py 
# SSE clarification 이벤트 전송
yield (
    "event: clarification\n"
    f"data: {json.dumps({
        'type': 'clarification',
        'question': clarification_question,
        'options': clarification_options,
        'session_id': session_id
    })}\n\n"
)

Orchestrator LangGraph 구조

전체 흐름은 OrchestratorState 기반 LangGraph StateGraph로 구성됩니다:

  • analyze_ambiguity → 조건 분기: build_clarification | pass_through | resolve_clarification
  • 사용자가 선택지를 클릭하면 Redis에 저장된 pending clarification을 불러와 resolve_clarification 노드에서 원래 질문 + 선택 답변을 합성합니다.
Q4가드레일(guardrail) 시스템은 어떻게 동작하나? (LangGraph 상태머신, 하드블록 + LLM 분류)+

가드레일 시스템은 2단계 방어 구조로, 사용자 입력이 튜터의 서비스 범위에 해당하는지 검사합니다. 첫 번째는 키워드 기반 하드 차단, 두 번째는 LLM 기반 의미 분류입니다.

1단계: 하드 차단 (키워드 필터)

chatbot/services/guardrail.py 
MALICIOUS_KEYWORDS = [
    "씨발", "개새끼", "병신", "지랄",
    "탈옥", "system prompt", "ignore previous instructions",
    "forget previous instructions"
]

async def run_guardrail(message: str, context: str = "") -> GuardrailResult:
    # 1단계: 하드 차단 키워드 필터 (MALICIOUS)
    lower_msg = message.lower()
    for kw in MALICIOUS_KEYWORDS:
        if kw in lower_msg:
            return GuardrailResult(
                False,
                "부적절하거나 안전하지 않은 요청이 감지되었습니다.",
                "MALICIOUS"
            )

2단계: LLM 분류 (LangGraph StateGraph)

하드 차단을 통과한 메시지는 GuardrailState 기반 LangGraph로 전달됩니다. classify_input 노드에서 gpt-4o-mini(temperature=0)가 4가지 카테고리로 분류합니다:

분류설명처리
SAFE정상 금융 질문, 인사말통과
ADVICE투자 자문 (매수/매도 추천)정책에 따라 차단 또는 통과
OFF_TOPIC금융 무관 질문 (맛집, 연애 등)차단
MALICIOUS프롬프트 인젝션, 욕설차단

LangGraph 워크플로우

workflow = StateGraph(GuardrailState)
workflow.add_node("classify_input", classify_input)
workflow.add_node("increment_retry", increment_retry)

workflow.set_entry_point("classify_input")
workflow.add_conditional_edges(
    "classify_input",
    decide_route,
    {"retry": "increment_retry", "end": END}
)
workflow.add_edge("increment_retry", "classify_input")

JSON 파싱 실패 시 최대 MAX_RETRIES=2회까지 재시도합니다. increment_retry 노드에서 retries 카운터를 증가시키고 다시 classify_input으로 돌아갑니다. API 장애 시에는 Fail-open 정책으로 SAFE를 반환하여 서비스 가용성을 보장합니다.

가드레일 프롬프트는 chatbot/prompts/templates/guardrail.md에 6가지 Few-Shot 판례와 함께 정의되어 있어, LLM이 경계 사례를 정확히 판단할 수 있도록 합니다.

Q5가드레일 정책(soft/hard)의 차이는? (TUTOR_GUARDRAIL_POLICY 환경변수)+

LLM 분류 후 최종 차단 여부는 TUTOR_GUARDRAIL_POLICY 환경변수에 의해 결정됩니다. 이 정책은 특히 ADVICE 판정(투자 자문 요청)에 대한 대응을 제어합니다.

chatbot/services/guardrail.py 
# 2단계 정책 설정
policy = os.getenv("TUTOR_GUARDRAIL_POLICY", "soft")

is_allowed = (decision == "SAFE")
block_message = ""

if policy == "soft" and decision == "ADVICE":
    # 유연화 정책: ADVICE 판정시 대화 흐름을 끊지 않고
    # 챗봇이 우회 교육답변을 하도록 통과
    is_allowed = True
    block_message = ""
elif decision == "ADVICE":
    block_message = (
        "죄송합니다만, 특정 종목에 대한 투자 자문(매수/매도 추천 등)은 "
        "자본시장법상 제공해 드릴 수 없습니다. "
        "기업의 객관적인 재무 지표나 시장 동향에 대해서라면 답변해 드릴 수 있어요!"
    )
정책SAFEADVICEOFF_TOPICMALICIOUS
soft (기본값)통과통과 (교육적 우회 답변)차단차단
hard통과차단 (자본시장법 안내)차단차단

soft 정책은 프로덕션 기본값으로, 투자 자문 의도의 질문이 들어와도 대화 흐름을 끊지 않습니다. 대신 LLM 시스템 프롬프트의 "단정적 조언 금지" 규칙에 의해 챗봇이 자연스럽게 교육적 관점으로 우회 답변합니다. 예를 들어 "삼성전자 사야 해?"라는 질문에 직접적인 매수 추천 대신 PER, 업종 비교 등 객관적 지표를 안내합니다.

hard 정책은 자본시장법 준수가 엄격히 요구되는 환경에서 사용합니다. ADVICE로 분류된 질문은 즉시 차단 메시지로 응답하고 LLM 호출 자체를 생략합니다.

OFF_TOPIC과 MALICIOUS는 정책과 무관하게 항상 차단됩니다. PARSE_ERROR(JSON 파싱 재시도 실패) 시에는 일시적 오류 메시지를 반환합니다.

Q6SSE 스트리밍 구현 방법은? (이벤트 타입과 엔드포인트)+

튜터 채팅은 FastAPI의 StreamingResponse와 Server-Sent Events (SSE)를 사용하여 실시간 스트리밍됩니다. 엔드포인트는 POST /api/v1/tutor/chat이며, 분당 10회 레이트 리밋이 적용됩니다.

fastapi/app/api/routes/tutor.py 
@router.post("/chat")
@limiter.limit("10/minute")
async def tutor_chat(
    request: Request,
    chat_request: TutorChatRequest,
    db: AsyncSession = Depends(get_db),
    current_user: Optional[dict] = Depends(get_current_user_optional),
) -> StreamingResponse:
    return StreamingResponse(
        generate_tutor_response_stream(chat_request, db, request, current_user),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",  # nginx 버퍼링 비활성화
        },
    )

SSE 이벤트 타입 (TutorChatEvent 스키마)

fastapi/app/schemas/tutor.py 
class TutorChatEvent(BaseModel):
    type: Literal[
        "thinking",        # 분석 중 표시
        "tool_call",       # 도구 호출 알림
        "text_delta",      # 텍스트 청크 스트리밍
        "done",            # 완료 (session_id, sources, total_tokens 포함)
        "error",           # 에러 발생
        "visualization",   # Plotly JSON 차트 데이터
        "viz_intent",      # 차트 생성 시작 안내
        "clarification",   # 확인 질문 + 선택지 버튼
    ]

이벤트 전송 순서 (일반 흐름)

  1. event: step (type: thinking) - "질문을 분석하고 있습니다..."
  2. event: viz_intent - 차트 생성 안내 (시각화 시)
  3. event: visualization - Plotly JSON 차트 데이터 (시각화 시)
  4. event: text_delta - LLM 응답 텍스트 청크 (반복)
  5. event: done - 완료 (session_id, sources, total_tokens)

분기 흐름

  • Clarification 흐름: 모호한 질문 감지 시 event: clarification → event: done으로 즉시 종료
  • 가드레일 차단: event: text_delta (차단 메시지) → event: done (guardrail 필드 포함)
  • 에러: event: error 단독 전송

스트리밍 타임아웃은 300초이며, 10청크마다 클라이언트 연결 해제를 감지하여 조기 중단합니다.

Q11메시지 윈도잉은 어떻게 하나? (MAX_MESSAGES=40, 최근 20턴 유지)+

튜터 에이전트는 두 계층에서 메시지 윈도잉을 수행합니다: LangGraph 에이전트 노드에서의 인메모리 윈도잉, 그리고 DB에서 이전 메시지를 로드할 때의 제한입니다.

1. LangGraph 에이전트 노드: MAX_MESSAGES=40

chatbot/agent/tutor_agent.py 
def _agent_node(self, state: AgentState) -> dict:
    messages = list(state["messages"])
    difficulty = state.get("difficulty", self.default_difficulty)

    # Window memory: keep max 20 turns (40 messages)
    MAX_MESSAGES = 40  # 20 turns (user + assistant = 2 messages per turn)
    if len(messages) > MAX_MESSAGES:
        messages = messages[-MAX_MESSAGES:]

    system_prompt = get_system_prompt(difficulty)
    all_messages = [SystemMessage(content=system_prompt)] + messages
    response = self.llm_with_tools.invoke(all_messages)
    return {"messages": [response]}

LangGraph StateGraph의 add_messages reducer는 모든 메시지를 누적하지만, _agent_node에서 LLM에 전달할 때 최근 40개 메시지(20턴)만 슬라이싱합니다. 이는 gpt-4o-mini의 컨텍스트 윈도우를 효율적으로 사용하면서 대화 맥락을 유지하기 위한 설계입니다.

  • 컨텍스트 품질: 오래된 메시지보다 최근 대화가 더 관련성 높은 맥락을 제공

Q12모호성 탐지 휴리스틱은? (대명사, 모호한 동사, 짧은 메시지)+

Hybrid Clarification Flow의 첫 번째 단계인 휴리스틱 모호성 탐지는 LLM 호출 없이 빠르게 모호한 질문을 걸러냅니다. 규칙 기반으로 대명사, 모호한 동사, 메시지 길이를 조합하여 판단합니다.

chatbot/services/tutor_orchestrator_graph.py 
def _heuristic_ambiguity(message: str) -> tuple[bool, list[str], str]:
    cleaned = message.strip()
    if not cleaned:
        return True, ["topic"], "입력이 비어 있음"

    # 대명사/지시어 패턴
    pronoun_like = ("이거", "저거", "그거", "이 내용", "이 화면", "방금", "그냥")
    # 모호한 동사 패턴
    vague_verbs = ("해줘", "알려줘", "보여줘", "설명해줘", "자세히")

    has_pronoun = any(token in cleaned for token in pronoun_like)
    has_vague_verb = any(token in cleaned for token in vague_verbs)

    # 규칙 1: 8자 이하 + (대명사 또는 모호한 동사)
    if len(cleaned) <= 8 and (has_pronoun or has_vague_verb):
        return True, ["topic"], "짧고 지시 대상이 불명확함"

    # 규칙 2: 대명사 + 모호한 동사 동시 존재
    if has_pronoun and has_vague_verb:
        return True, ["topic"], "지시 대상/범위가 불명확함"

    return False, [], "명확한 요청"

기간 관련 슬롯 탐지

모호성이 감지된 후, missing_slots에 따라 어떤 선택지를 제시할지 결정하는 헬퍼 함수도 있습니다:

def _is_period_related(message: str, missing_slots: list[str]) -> bool:
    lower = message.lower()
    period_keywords = ("기간", "언제", "최근", "시점", "range", "period", "월", "주", "일")
    if any(slot in {"period", "time_range", "date_range"} for slot in missing_slots):
        return True
    return any(keyword in lower for keyword in period_keywords)

기간 관련으로 판별되면 PERIOD_OPTIONS(10거래일/1개월/3개월)을, 아니면 GENERAL_OPTIONS(현재 화면 기준/특정 종목 기준/최근 데이터 기준)을 기본 선택지로 제공합니다.

Q13컨텍스트 수집은 어떤 순서로 하나?+

튜터 엔진은 LLM에 전달할 동적 컨텍스트를 5가지 소스에서 수집합니다. 수집 순서는 다음과 같으며, 각 단계가 실패해도 다음 단계로 진행됩니다 (graceful degradation).

컨텍스트 수집 순서

chatbot/services/tutor_engine.py (generate_tutor_response_stream)

1단계: 페이지 컨텍스트 (page_context)

사용자가 현재 보고 있는 화면 정보를 수집합니다. 두 가지 경로가 있습니다:

  • context_text: 프론트엔드가 직접 전송한 화면 텍스트 (최우선)
  • context_type + context_id: DB에서 브리핑/사례를 조회
if request.context_text:
    page_context = (
        "\n\n[현재 학습/화면 문맥]\n"
        "사용자는 지금 애플리케이션 화면에서 아래의 내용을 직접 읽고 있는 중입니다:\n"
        f'"""\n{request.context_text}\n"""'
    )

Q14튜터 세션 관리는? (UUID 기반, anonymous 지원, Redis 캐시)+

튜터 세션은 UUID 기반으로 관리되며, 인증 사용자와 익명 사용자 모두 지원합니다. 세션 데이터는 PostgreSQL에 영구 저장되고, Redis에 캐싱되어 빠른 조회가 가능합니다.

세션 생성 (_ensure_session)

chatbot/services/tutor_engine.py 
async def _ensure_session(db, request, session_id, current_user) -> TutorSession:
    # 기존 세션 조회
    if request.session_id:
        existing = await db.execute(
            select(TutorSession).where(
                TutorSession.session_uuid == uuid.UUID(request.session_id)
            )
        )
        session_obj = existing.scalar_one_or_none()

    if not session_obj:
        # 신규 세션 생성
        user_id = current_user["id"] if current_user else None  # 익명 지원
        session_obj = TutorSession(
            session_uuid=uuid.UUID(session_id),
            user_id=user_id,  # None이면 anonymous
            context_type=request.context_type,
            context_id=request.context_id,
            title=request.message[:50],
            message_count=0,
        )
        db.add(session_obj)
        await db.flush()

Anonymous 지원

채팅 엔드포인트(/tutor/chat)는 get_current_user_optional을 사용하여 JWT 토큰이 없어도 동작합니다. 미인증 시 user_id=None으로 세션이 생성되며, 포트폴리오 컨텍스트는 생략됩니다. 세션 목록 조회 등 CRUD API는 인증이 필수입니다.

Clarification 상태 관리

모호성 확인 질문의 상태도 Redis에 저장됩니다. TTL은 15분(CLARIFICATION_TTL_SECONDS=900)이며, 만료된 clarification은 자동 폐기됩니다. 키 형식은 tutor:clarify:{session_id}입니다.

CLARIFICATION_KEY_PREFIX = "tutor:clarify:"
CLARIFICATION_TTL_SECONDS = 15 * 60

# 저장
await cache.set(key, json.dumps(payload), ttl=CLARIFICATION_TTL_SECONDS)

# 조회 시 KST 기반 TTL 이중 검증
created_at = datetime.fromisoformat(created_at_raw)
if datetime.now(KST) - created_at > timedelta(seconds=CLARIFICATION_TTL_SECONDS):
    await cache.delete(key)  # 만료 → 삭제
    return None

세션 삭제 시 Redis의 메시지 캐시와 clarification 캐시가 함께 제거됩니다.
