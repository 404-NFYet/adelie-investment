Q1PostHog 자체 호스팅 선택 이유와 구성은?+

PostHog를 SaaS가 아닌 자체 호스팅으로 운영하는 이유는 크게 세 가지입니다.

자체 호스팅 선택 이유

  • 데이터 주권: 금융 교육 플랫폼 특성상 사용자 행동 데이터(포트폴리오 조회, 거래 패턴 등)가 민감할 수 있어, 외부 서버로 전송하지 않고 내부에서 관리
  • 비용 절감: PostHog Cloud는 이벤트 볼륨에 따라 과금되지만, 자체 호스팅은 서버 비용만 발생하므로 데모/학습 플랫폼 환경에서 경제적
  • 커스터마이징 자유도: 세션 레코딩, 히트맵, 피처 플래그 등 전체 기능을 제한 없이 활용 가능

Q2MS Clarity 히트맵/세션 리플레이 활용은?+

Microsoft Clarity는 PostHog와 병행하여 히트맵과 세션 리플레이에 특화된 분석을 제공합니다. PostHog가 이벤트 기반 분석에 강점이 있다면, Clarity는 시각적 UX 분석에 최적화되어 있어 상호 보완적으로 사용합니다.

활용 영역

  • 히트맵: 키워드 카드, 내러티브 스텝, 모의투자 버튼 등 모바일 UI 요소의 실제 터치/클릭 분포 분석
  • 세션 리플레이: 사용자가 내러티브 7단계를 어느 단계에서 이탈하는지, 튜터 채팅 진입 경로가 어디인지 시각적으로 확인
  • Rage Click 감지: 반응이 없는 UI 요소를 반복 터치하는 패턴을 자동으로 감지하여 UX 문제점 파악

초기화 방식

Clarity SDK는 @microsoft/clarity npm 패키지를 dynamic import하여 로드합니다. VITE_CLARITY_ID 환경변수가 없으면 초기화를 건너뛰어, 개발 환경에서는 Clarity가 작동하지 않습니다.

frontend/src/utils/analyticsProviders.js:18-27 
export async function initClarity() {
  if (!CLARITY_ID) return;
  try {
    const { default: Clarity } = await import('@microsoft/clarity');
    Clarity.init(CLARITY_ID);
    clarityReady = true;
  } catch (err) {
    console.warn('[Analytics] Clarity 초기화 실패:', err);
  }
}
Q3프론트엔드 이벤트 트래킹 구조는?+

이벤트 트래킹은 Bridge 패턴으로 설계되어 있습니다. 하나의 trackEvent() 호출이 최대 3개 목적지로 동시 전송됩니다.

3계층 전송 구조

목적지역할메커니즘
자체 DB상세 이벤트 저장배치 전송 (/api/v1/analytics/events)
Clarity히트맵/세션 리플레이clarityEvent() fire-and-forget
PostHog이벤트 분석/퍼널posthogEvent() fire-and-forget

배치 전송 메커니즘

자체 DB로의 전송은 배치 처리를 사용합니다. 이벤트를 큐에 쌓아두고, 10개 도달 시 즉시 또는 5초 타이머 만료 시 일괄 전송합니다. 전송 실패 시에는 큐에 재추가하되, 최대 100개로 제한하여 메모리 누수를 방지합니다.

frontend/src/utils/analytics.js:45-72 
export function trackEvent(eventType, data = {}) {
  const event = {
    user_id: getCurrentUserId(),
    session_id: getSessionId(),
    event_type: eventType,
    event_data: {
      ...data,
      timestamp: Date.now(),
      page: window.location.pathname,
    },
  };

  eventQueue.push(event);

  // 외부 분석 도구로 전달 (fire-and-forget)
  clarityEvent(eventType);
  posthogEvent(eventType, { ...data, page: window.location.pathname });

  // 배치 사이즈 도달 시 즉시 전송
  if (eventQueue.length >= BATCH_SIZE) {
    flushEvents();
  }

  // 타이머 설정 (아직 없으면)
  if (!flushTimer) {
    flushTimer = setTimeout(flushEvents, FLUSH_INTERVAL);
  }
}

주요 이벤트 유형

  • page_view, page_duration — 페이지 조회 및 체류 시간
  • narrative_step — 내러티브 단계 진행
  • trade_execute — 모의투자 거래 실행
  • tutor_ask, tutor_selection_submit — AI 튜터 질문
  • term_click — 용어 클릭
  • narrative_selection_cta_exposed, narrative_selection_ask_click — CTA 노출/클릭

페이지 이탈 시(beforeunload) 또는 탭이 비활성화될 때(visibilitychange) 즉시 flush하여 데이터 유실을 최소화합니다.

frontend/src/utils/analytics.js:124-131 
if (typeof window !== 'undefined') {
  window.addEventListener('beforeunload', flushEvents);
  window.addEventListener('visibilitychange', () => {
    if (document.visibilityState === 'hidden') {
      flushEvents();
    }
  });
}
Q4PostHog 사용자 식별(identify)은 어떻게 하나?+

PostHog의 사용자 식별은 AnalyticsProvider 컴포넌트가 UserContext의 user 상태를 감시하여 자동으로 처리합니다.

식별 흐름

  1. 로그인 시: user.isAuthenticated가 true가 되면 identifyUser(user) 호출
  2. PostHog identify: posthogInstance.identify(String(user.id), { email, nickname })로 사용자 ID를 PostHog에 등록
  3. 로그아웃 시: resetUser()가 posthogInstance.reset()을 호출하여 익명 상태로 복귀

이 패턴 덕분에 로그인 전 익명 이벤트와 로그인 후 이벤트가 자동으로 병합(merge)되어, 한 사용자의 전체 여정을 추적할 수 있습니다.

frontend/src/utils/analyticsProviders.js:55-69 
export function identifyUser(user) {
  if (!user?.id) return;
  try {
    if (clarityReady && window.clarity) {
      window.clarity('identify', String(user.id));
    }
  } catch {}
  try {
    if (posthogInstance) {
      posthogInstance.identify(String(user.id), {
        email: user.email || undefined,
        nickname: user.nickname || undefined,
      });
    }
  } catch {}
}
frontend/src/components/common/AnalyticsProvider.jsx:30-37 
// 유저 식별 (로그인/로그아웃)
useEffect(() => {
  if (user?.isAuthenticated) {
    identifyUser(user);
  } else {
    resetUser();
  }
}, [user]);

Q5Clarity 이벤트 추적은 어떻게 하나?+

Clarity는 PostHog와 달리 커스텀 속성(properties) 전달이 불가능하므로, 이벤트 이름만 전송하는 단순한 인터페이스를 사용합니다.

커스텀 이벤트 (clarityEvent)

window.clarity('event', eventType)을 호출하여 Clarity 대시보드에서 특정 이벤트가 발생한 세션을 필터링할 수 있게 합니다.

frontend/src/utils/analyticsProviders.js:86-91 
export function clarityEvent(eventType) {
  if (!clarityReady || !window.clarity) return;
  try {
    window.clarity('event', eventType);
  } catch {}
}

SPA 페이지뷰 (clarityPageView)

React Router의 location.pathname 변경을 감지하여 page_view 이벤트를 Clarity에 전송합니다. Clarity는 기본적으로 SPA 라우트 변경을 자동 감지하지만, 명시적 이벤트도 전송하여 정확성을 보장합니다.

frontend/src/utils/analyticsProviders.js:106-112 
export function clarityPageView() {
  if (!clarityReady || !window.clarity) return;
  try {
    // Clarity는 SPA 변경을 자동 감지하지만, 명시적 이벤트도 전송
    window.clarity('event', 'page_view');
  } catch {}
}

사용자 식별

Clarity의 identify는 PostHog와 동일한 시점에 호출되며, user.id를 문자열로 전달합니다. 이를 통해 Clarity 대시보드에서 특정 사용자의 세션 리플레이를 찾을 수 있습니다.

window.clarity('identify', String(user.id));

Q7PostHog 자동 캡처(autocapture) 설정은?+

PostHog의 autocapture 기능은 true로 활성화되어 있어, 별도의 코드 계측 없이도 사용자의 클릭, 폼 입력, 페이지 이동 등을 자동으로 수집합니다.

설정 내용

frontend/src/utils/analyticsProviders.js:36-46 
posthog.init(POSTHOG_KEY, {
  api_host: POSTHOG_HOST,
  capture_pageview: false,   // SPA — 수동 페이지뷰
  autocapture: true,         // 클릭/폼 자동 수집
  persistence: 'localStorage',
  session_recording: {
    maskAllInputs: true,
    maskTextSelector: '[data-ph-mask]',
  },
});
posthogInstance = posthog;

주요 설정 항목 해설

옵션값이유
capture_pageviewfalseSPA이므로 React Router 변경 시 수동으로 $pageview 캡처 (자동 캡처 시 초기 1회만 발생)
autocapturetrue버튼 클릭, 링크 클릭, 폼 제출 등 사용자 인터랙션을 자동으로 수집
persistencelocalStorage쿠키 대신 localStorage에 distinct_id 저장 (쿠키 동의 불필요)

수동 페이지뷰의 동작 방식

AnalyticsProvider가 location.pathname 변경을 감지하면 posthogPageView()를 호출하여 PostHog에 $pageview 이벤트를 전송합니다.

frontend/src/utils/analyticsProviders.js:117-122 
export function posthogPageView() {
  if (!posthogInstance) return;
  try {
    posthogInstance.capture('$pageview');
  } catch {}
}

이렇게 하면 / → /narrative/123 → /education 같은 SPA 내비게이션이 모두 정확히 추적됩니다.

Q8세션 레코딩 + 입력 마스킹 설정은?+

PostHog의 세션 레코딩은 사용자의 화면 인터랙션을 녹화하여, 실제 사용 패턴을 시각적으로 분석할 수 있게 합니다. 그러나 금융 교육 플랫폼 특성상 개인정보 보호가 중요하므로 입력 마스킹을 적극 적용합니다.

마스킹 설정

frontend/src/utils/analyticsProviders.js:41-44 
session_recording: {
  maskAllInputs: true,
  maskTextSelector: '[data-ph-mask]',

maskAllInputs: true모든 <input>, <textarea> 입력값이 녹화에서 '***'로 대체됨. 비밀번호, 이메일, 거래 금액 등 민감 정보 보호

Clarity와의 역할 분담

  • PostHog 세션 레코딩: 특정 사용자/이벤트 기반 필터링이 강력. 예: "퀴즈 오답 후 이탈한 사용자" 세션만 조회
