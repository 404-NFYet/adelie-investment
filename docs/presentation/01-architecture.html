Q2 프론트엔드와 백엔드 간 통신 방식은? (nginx 리버스 프록시, /api/v1 prefix) +

프론트엔드와 백엔드 간 통신은 nginx 리버스 프록시를 통해 이루어집니다. React SPA와 FastAPI가 동일한 도메인(demo.adelie-invest.com)에서 서비스되며, 경로 기반으로 라우팅이 분리됩니다.

  }
  return response.json();
};

인증 흐름

인증은 HttpOnly 쿠키 + CSRF Double-Submit 패턴으로 처리됩니다. authFetch는 credentials: 'include'로 쿠키를 자동 전송하고, POST/PUT/DELETE 요청 시 X-CSRF-Token 헤더를 자동 추가합니다. 401 응답 시 refresh 토큰으로 자동 갱신 후 1회 재시도합니다.

frontend/src/api/client.js:31-45 
/** CSRF 토큰 쿠키 읽기 (Double-Submit Cookie 패턴) */
const getCsrfToken = () => {
  const match = document.cookie.match(/(?:^|;\s*)csrfToken=([^;]*)/);
  return match ? decodeURIComponent(match[1]) : '';
};

/** 요청 헤더 생성: POST/PUT/PATCH/DELETE 시 CSRF 토큰 포함 */
const mergeHeaders = (headers, method) => {
  const merged = new Headers(headers || {});
  if (!['GET', 'HEAD', 'OPTIONS'].includes(method?.toUpperCase())) {
    const csrf = getCsrfToken();
    if (csrf) merged.set('X-CSRF-Token', csrf);
  }
  return merged;
};

또한 nginx는 SPA 라우팅을 위해 try_files $uri $uri/ /index.html을 설정하여, React Router의 클라이언트 사이드 라우팅이 정상 동작하도록 합니다. 정적 자산(js, css, 이미지)에는 7일 캐시를 적용하고, index.html과 PWA 관련 파일(sw.js, manifest.webmanifest)에는 캐시 방지 헤더를 설정합니다.

Q3 모놀리식 vs 마이크로서비스 -- 왜 현재 구조를 선택했나? +

Adelie Investment는 "모듈러 모놀리스(Modular Monolith)" 구조를 채택했습니다. 단일 FastAPI 프로세스가 인증, API, 튜터 챗봇, 스케줄러를 모두 처리하지만, 코드베이스는 fastapi/, chatbot/, datapipeline/으로 명확히 분리되어 있습니다.

구조적 특징

특징모놀리식 요소마이크로서비스 요소
배포 단위backend-api 하나의 컨테이너frontend, pipeline은 별도 컨테이너
코드 분리하나의 프로세스에서 실행chatbot/, datapipeline/ 독립 모듈
DB 접근모든 모듈이 같은 PostgreSQL 사용-
통신 방식함수 호출 (in-process)HTTP 프록시 (nginx → FastAPI)

이 구조를 선택한 이유

  1. 팀 규모 적합성: 5명의 소규모 팀에서 마이크로서비스의 운영 오버헤드(서비스 디스커버리, 분산 트레이싱, 개별 배포 파이프라인)는 비효율적입니다.
  2. 데이터 일관성: 키워드, 케이스, 사용자 데이터가 강하게 결합되어 있어 단일 DB가 트랜잭션 일관성을 보장합니다.
  3. 개발 속도: 모듈 간 함수 호출로 통신하므로 API 계약 관리, 직렬화 오버헤드가 없습니다.
  4. 향후 분리 가능: chatbot/과 datapipeline/이 이미 독립된 Python 패키지로 분리되어 있어, 필요시 별도 서비스로 추출할 수 있습니다.

Docker Compose 프로덕션 설정에서 ai-pipeline 서비스가 profiles: [pipeline]으로 분리되어 있는 것이 이 하이브리드 접근의 증거입니다. 파이프라인은 on-demand 실행 시에만 별도 컨테이너로 기동되고, 일상적인 스케줄 실행은 backend-api 프로세스 내부에서 subprocess로 처리됩니다.

Q4 서비스 간 의존성은 어떻게 관리하나? (FastAPI 동적 라우터 등록, importlib) +

FastAPI 백엔드는 동적 라우터 등록(Dynamic Router Registration) 패턴을 사용합니다. importlib을 통해 18개의 라우트 모듈을 개별적으로 import하고, 실패한 모듈은 경고 로그만 남기고 건너뜁니다. 이로써 Docker 환경에서 일부 모듈이 존재하지 않아도 서비스가 기동됩니다.

FastAPI 백엔드가 chatbot/과 datapipeline/ 모듈을 사용하는 방식은 크게 2가지 패턴으로 나뉩니다: (1) Docker 빌드 시 소스 복사 + 직접 import, (2) sys.path 조작을 통한 런타임 import.

Q6 SSE 스트리밍은 왜 WebSocket 대신 선택했나? +

AI 튜터 챗봇의 실시간 응답 전달에 SSE(Server-Sent Events)를 선택한 이유는 크게 4가지입니다.

1. 단방향 스트리밍에 최적화

튜터 챗봇은 사용자가 질문을 POST로 보내고, 서버가 LLM 응답을 토큰 단위로 스트리밍하는 요청-스트림 패턴입니다. 양방향 통신이 필요 없으므로 WebSocket의 복잡성(핸드셰이크, 연결 관리, 하트비트)이 불필요합니다.

2. HTTP/1.1 기반의 단순한 인프라 호환

SSE는 일반 HTTP 응답이므로 nginx 리버스 프록시, Cloudflare Tunnel 등 기존 인프라를 그대로 사용할 수 있습니다. WebSocket은 Upgrade 핸드셰이크가 필요하고 프록시 설정이 별도로 필요합니다.

Q7 API 응답 공통 포맷(envelope)은 어떻게 구현했나? +

모든 API 응답은 미들웨어 레벨의 자동 래핑(Response Envelope Middleware)을 통해 일관된 JSON 구조로 변환됩니다. 각 라우트 핸들러가 개별적으로 래핑할 필요 없이, 미들웨어가 전역적으로 처리합니다.

응답 포맷 규격

상태구조예시
성공 (2xx){"status": "success", "data": ...}정상 API 응답
에러 (4xx/5xx){"status": "error", "message": "...", "error": ...}HTTP/Validation 에러

구현: response_envelope_middleware

fastapi/app/main.py:230-261 
@app.middleware("http")
async def response_envelope_middleware(request: Request, call_next):
    """응답을 전역 스키마로 래핑 (JSON만 적용)."""
    response = await call_next(request)

    # 문서/메트릭/스키마/스트리밍 응답은 제외
    if request.url.path in {"/metrics", "/docs", "/redoc", "/openapi.json"}:
        return response
    if isinstance(response, StreamingResponse):
        return response

    content_type = response.headers.get("content-type", "")
    if "application/json" not in content_type:
        return response

    try:
        body = response.body
        payload = json.loads(body.decode("utf-8")) if body else None
    except Exception:
        return response

    # 이미 전역 포맷이면 그대로 반환
    if isinstance(payload, dict) and payload.get("status") in {"success", "error"}:
        return response

    if response.status_code >= 400:
        wrapped = {"status": "error", "message": "Request failed", "error": payload}
    else:
        wrapped = {"status": "success", "data": payload}
    headers = dict(response.headers)
    headers.pop("content-length", None)
    return JSONResponse(content=wrapped, status_code=response.status_code, headers=headers)

래핑 제외 대상

  • 문서/메트릭 경로: /metrics, /docs, /redoc, /openapi.json
  • StreamingResponse: SSE 스트리밍 (튜터 챗봇)
  • 비-JSON 응답: HTML, 이미지 등
  • 이미 래핑된 응답: payload.get("status") in {"success", "error"} 체크로 이중 래핑 방지

이 설계의 핵심은 라우트 핸들러의 책임 분리입니다. 각 핸들러는 순수한 데이터만 반환하면 되고, 에러 핸들러와 미들웨어가 일관된 형식을 보장합니다. 예외 핸들러에서 반환하는 응답도 "status": "error"를 포함하므로 이중 래핑되지 않습니다.

Q8 글로벌 예외 처리는 어떻게 동작하나? (3단계: Global → HTTP → Validation) +

FastAPI 백엔드는 3단계 예외 처리 계층을 구현합니다. 각 계층이 서로 다른 유형의 예외를 처리하며, 모든 예외가 일관된 JSON 형식으로 클라이언트에 전달됩니다.

2단계: HTTPException (4xx/5xx)

비즈니스 로직에서 명시적으로 발생시킨 HTTP 에러를 처리합니다. detail이 dict인 경우 구조를 유지하고, 문자열인 경우 메시지로 변환합니다.

fastapi/app/main.py:137-154 
@app.exception_handler(HTTPException)
async def http_exception_handler(request: Request, exc: HTTPException):
    """HTTP 예외를 전역 에러 포맷으로 변환."""
    detail = exc.detail
    if isinstance(detail, dict):
        message = detail.get("message") or detail.get("error") or "Request failed"
        error = {k: v for k, v in detail.items() if k != "message"}
    else:
        message = str(detail) if detail else "Request failed"
        error = None
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "status": "error",
            "message": message,
            "error": error,
        },
    )

Rate Limit 미들웨어 상세

Redis의 Lua 스크립트를 사용한 원자적 슬라이딩 윈도우 방식으로 구현됩니다. Redis 장애 시에는 레이트리밋을 적용하지 않고 요청을 허용합니다(graceful degradation).

fastapi/app/middleware/rate_limit.py:15-36 
SLIDING_WINDOW_LUA = """
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
local count = redis.call('ZCARD', key)

if count < limit then
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, math.ceil(window / 1000))
    return {1, limit - count - 1, 0}
else
    local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
    local reset = oldest[2] and (tonumber(oldest[2]) + window) or (now + window)
    return {0, 0, reset}
end
"""

RATE_LIMIT = 100       -- 요청/분
WINDOW_MS = 60_000     -- 1분 (ms)

CSRF 미들웨어 상세

CSRF 보호는 Double-Submit Cookie 패턴을 사용합니다. GET/HEAD/OPTIONS는 면제, 인증 경로(/api/v1/auth/login 등)도 면제됩니다. CSRF 쿠키가 없는 요청(Authorization 헤더 방식)은 검증을 건너뜁니다 -- Authorization 헤더 방식은 CSRF에 취약하지 않기 때문입니다.

fastapi/app/main.py:199-227 
@app.middleware("http")
async def csrf_middleware(request: Request, call_next):
    settings = get_settings()
    if request.method in {"GET", "HEAD", "OPTIONS"}:
        return await call_next(request)

    path = request.url.path
    csrf_exempt_prefixes = {
        "/api/v1/auth/login",
        "/api/v1/auth/register",
        "/api/v1/auth/refresh",
        "/api/v1/auth/csrf",
    }
    if path in csrf_exempt_prefixes:
        return await call_next(request)

    csrf_cookie = request.cookies.get(settings.AUTH_CSRF_COOKIE_NAME)
    if not csrf_cookie:
        # CSRF 쿠키 없음 = 쿠키 인증 미사용 클라이언트 → 검증 스킵
        return await call_next(request)
    csrf_header = request.headers.get(settings.AUTH_CSRF_HEADER_NAME)
    if not csrf_header or csrf_cookie != csrf_header:
        return JSONResponse(
            status_code=403,
            content={"detail": "CSRF token missing or invalid"},
        )
    return await call_next(request)

  3. 프론트엔드 도메인 분리: src/api/가 도메인별 파일(auth.js, portfolio.js, quiz.js, learning.js)로 분리되어, 각 API 호출이 어떤 도메인에 속하는지 명확합니다.
  4. 인프라 코드 분리: infra/, lxd/는 애플리케이션 코드와 완전히 분리됩니다.

핵심 설계 결정

  • SSL 종단을 Cloudflare에 위임: 서버에서 직접 Let's Encrypt 인증서를 관리하지 않아도 됩니다. Cloudflare가 자동으로 SSL을 처리합니다.
  • 포트 노출 최소화: 서버의 방화벽에서 80, 3000, 8501 등의 포트를 외부에 직접 노출하지 않습니다. Cloudflare Tunnel이 outbound 연결로 동작하므로 inbound 포트 오픈이 불필요합니다.
  • DDoS 보호: Cloudflare의 L3/L4/L7 DDoS 방어가 자동으로 적용됩니다.
  • Dashboard 이중 프록시: dashboard.adelie-invest.com은 Cloudflare Tunnel → nginx-dashboard(:8501) → Streamlit(:8501 내부)의 2단 프록시 구조입니다. nginx가 Streamlit의 _stcore 경로를 재작성하는 역할을 합니다.
